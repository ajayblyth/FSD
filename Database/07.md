======================================================================
MYSQL FOREIGN KEY (COMPLETE NOTES)
======================================================================

DEFINITION:
Foreign Key is a column (or set of columns) that refers to
the PRIMARY KEY of another table.

→ It creates a relationship between two tables

----------------------------------------------------------------------
WHY FOREIGN KEY IS USED
----------------------------------------------------------------------
• Maintain data integrity
• Prevent invalid data
• Establish relationships (parent-child)

----------------------------------------------------------------------
BASIC SYNTAX (INLINE)
----------------------------------------------------------------------


CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);

----------------------------------------------------------------------
SEPARATE CONSTRAINT SYNTAX
----------------------------------------------------------------------
CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT,
    CONSTRAINT fk_customer
    FOREIGN KEY (customer_id)
    REFERENCES customers(id)
);

----------------------------------------------------------------------
EXAMPLE TABLES
----------------------------------------------------------------------

CUSTOMERS (Parent Table)
--------------------------------
id | name
1  | Ajay
2  | Rahul

ORDERS (Child Table)
--------------------------------
id | customer_id
1  | 1
2  | 2

→ customer_id refers to customers.id

----------------------------------------------------------------------
RULES
----------------------------------------------------------------------
1) Referenced column must be PRIMARY KEY or UNIQUE
2) Data types must match
3) Cannot insert value not present in parent table
4) Parent row cannot be deleted (by default)
5) foreign key should always point to a unique column in a table.

----------------------------------------------------------------------
INSERT BEHAVIOR
----------------------------------------------------------------------
✔ VALID:
INSERT INTO orders(orders_date,customer_id) VALUES ('2026-03-03', 1);

❌ INVALID:
INSERT INTO orders(orders_date,customer_id)  VALUES ('2026-03-03', 10);
→ Error (10 not in customers) //10 not in parent table's id column

----------------------------------------------------------------------
ON DELETE / ON UPDATE ACTIONS
----------------------------------------------------------------------

1) CASCADE
----------------------------------------------------------------------
ON DELETE CASCADE

→ Deleting parent row deletes child rows

----------------------------------------------------------------------
2) SET NULL
----------------------------------------------------------------------
ON DELETE SET NULL

→ Sets foreign key to NULL

----------------------------------------------------------------------
3) RESTRICT (DEFAULT)
----------------------------------------------------------------------
→ Prevents delete if child exists

----------------------------------------------------------------------
4) NO ACTION
----------------------------------------------------------------------
→ Same as RESTRICT (in MySQL)

----------------------------------------------------------------------
FULL EXAMPLE
----------------------------------------------------------------------
CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT,
    FOREIGN KEY (customer_id)
    REFERENCES customers(id)
    ON DELETE CASCADE
    ON UPDATE CASCADE
);

----------------------------------------------------------------------
DROP FOREIGN KEY
----------------------------------------------------------------------
ALTER TABLE orders
DROP FOREIGN KEY fk_customer;

----------------------------------------------------------------------
CHECK FOREIGN KEYS
----------------------------------------------------------------------
SHOW CREATE TABLE orders;

----------------------------------------------------------------------
----------------------------------------------------------------------
EDGE CASES (WITH EXAMPLES)
----------------------------------------------------------------------

1) NULL VALUES
----------------------------------------------------------------------
Foreign key column can have NULL (if allowed)

CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);

INSERT INTO orders VALUES (1, NULL);
→ Works (NULL allowed, no parent check)

----------------------------------------------------------------------
----------------------------------------------------------------------
2) DELETE PARENT ROW
----------------------------------------------------------------------
Fails if child exists (default behavior)

-- Try deleting parent
DELETE FROM customers WHERE id = 1;

→ ERROR (if orders has customer_id = 1)

----------------------------------------------------------------------
✔ ON DELETE RESTRICT (DEFAULT)
----------------------------------------------------------------------
FOREIGN KEY (customer_id)
REFERENCES customers(id)
ON DELETE RESTRICT;

→ Prevents deletion of parent row if child exists

Example:
customers: id = 1
orders: customer_id = 1

DELETE FROM customers WHERE id = 1;

→ ERROR ❌ (deletion blocked)

----------------------------------------------------------------------
✔ ON DELETE CASCADE
----------------------------------------------------------------------
FOREIGN KEY (customer_id)
REFERENCES customers(id)
ON DELETE CASCADE;

→ Deletes child rows automatically

Example:
DELETE FROM customers WHERE id = 1;

→ Related orders also deleted

----------------------------------------------------------------------
✔ ON DELETE SET NULL
----------------------------------------------------------------------
CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT NULL,
    FOREIGN KEY (customer_id)
    REFERENCES customers(id)
    ON DELETE SET NULL
);

→ Keeps child row, removes reference

Example:
DELETE FROM customers WHERE id = 1;

→ orders.customer_id becomes NULL

----------------------------------------------------------------------
NOTES
----------------------------------------------------------------------
1) RESTRICT → block delete
2) CASCADE → delete child rows
3) SET NULL → remove reference only
4) Default behavior = RESTRICT

WHEN TO USE RESTRICT / CASCADE / SET NULL
======================================================================

RESTRICT:
→ Use when deletion should be blocked if related data exists  
→ Example: Prevent deleting a customer with active orders

CASCADE:
→ Use when child data should also be deleted automatically  
→ Example: Deleting user → delete posts, comments, dont use for payyments

SET NULL:
→ Use when relationship can be removed but data should remain  
→ Example: Order stays, but customer reference removed


======================================================================
CASCADE & PAYMENTS (REAL-WORLD RULE)
======================================================================

CASCADE:
→ Deletes child data automatically

GOOD USE:
→ Deleting user → delete posts, comments
→ Temporary / non-critical data

----------------------------------------------------------------------
❌ NOT RECOMMENDED FOR PAYMENTS
----------------------------------------------------------------------

Why?

1) Payments are CRITICAL financial records
2) Must be preserved for auditing & legal reasons
3) Accidental delete → permanent data loss ❌
4) History should NEVER disappear

----------------------------------------------------------------------
✔ WHAT TO USE INSTEAD
----------------------------------------------------------------------

• RESTRICT → prevent deletion if payment exists
• OR use soft delete (is_deleted flag)

Example:
→ Cannot delete user if payment exists

----------------------------------------------------------------------
FINAL RULE
----------------------------------------------------------------------
CASCADE → safe for non-critical data  
RESTRICT → use for important data (like payments)

======================================================================
----------------------------------------------------------------------
QUICK RULE
----------------------------------------------------------------------
• Strict integrity → RESTRICT  
• Dependent data → CASCADE  
• Optional relation → SET NULL  


======================================================================
----------------------------------------------------------------------
3) UPDATE PRIMARY KEY
----------------------------------------------------------------------
Fails unless ON UPDATE CASCADE used

-- Try updating parent key
UPDATE customers SET id = 5 WHERE id = 1;

→ ERROR (if child exists)

✔ With CASCADE:
ON UPDATE CASCADE

→ child table auto-updates:
customer_id: 1 → 5

----------------------------------------------------------------------
4) ENGINE REQUIREMENT
----------------------------------------------------------------------
Foreign keys work only with InnoDB

CREATE TABLE customers (
    id INT PRIMARY KEY
) ENGINE=MyISAM;

CREATE TABLE orders (
    id INT,
    customer_id INT,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
) ENGINE=MyISAM;

→ Foreign key IGNORED ❌ (no enforcement)

✔ Correct:
ENGINE=InnoDB

----------------------------------------------------------------------
NOTES
----------------------------------------------------------------------
1) NULL skips validation
2) CASCADE handles delete/update automatically
3) Without CASCADE → operations fail
4) Always use InnoDB for foreign keys

======================================================================
----------------------------------------------------------------------
COMMON MISTAKES
----------------------------------------------------------------------

❌ Mismatched data types
❌ Referencing non-primary/non-unique column
❌ Forgetting ON DELETE rules
❌ Using MyISAM engine

----------------------------------------------------------------------
NOTES
----------------------------------------------------------------------
1) Creates relationship between tables
2) Ensures referential integrity
3) Prevents invalid data insertion
4) Important for database design

MYSQL SUBQUERY (COMPLETE DETAILED NOTES)
======================================================================

DEFINITION:
A subquery is a query inside another query.

→ Inner query runs first, its result is used by outer query

----------------------------------------------------------------------
BASIC SYNTAX
----------------------------------------------------------------------
SELECT column_name
FROM table_name
WHERE column_name operator (
    SELECT column_name FROM table_name
);

----------------------------------------------------------------------
BASIC EXAMPLE
----------------------------------------------------------------------
SELECT name
FROM student
WHERE age > (
    SELECT AVG(age) FROM student
);

→ Students having age greater than average

• SELECT customer_id FROM customers where name = "John";
• SELECT * FROM orders where customer_id=(
    SELECT customer_id FROM customers where name = "John"
  );

----------------------------------------------------------------------
TYPES OF SUBQUERIES
----------------------------------------------------------------------

1) SINGLE VALUE (SCALAR)
----------------------------------------------------------------------
Returns one value

SELECT name FROM student
WHERE age = (SELECT MAX(age) FROM student);

----------------------------------------------------------------------
2) MULTIPLE VALUES
----------------------------------------------------------------------
Returns multiple values

SELECT name FROM student
WHERE city IN (
    SELECT city FROM branches
);

----------------------------------------------------------------------
3) MULTIPLE ROW / COLUMN
----------------------------------------------------------------------
SELECT * FROM student
WHERE (city, age) IN (
    SELECT city, age FROM other_table
);

----------------------------------------------------------------------
4) CORRELATED SUBQUERY
----------------------------------------------------------------------
Depends on outer query (runs per row)

SELECT name
FROM student s
WHERE age > (
    SELECT AVG(age)
    FROM student
    WHERE city = s.city
);

→ Compare with same city average

----------------------------------------------------------------------
SUBQUERY WITH SELECT
----------------------------------------------------------------------
SELECT name, (SELECT AVG(age) FROM student) AS avg_age
FROM student;

→ Adds extra column

----------------------------------------------------------------------
SUBQUERY WITH FROM (DERIVED TABLE)
----------------------------------------------------------------------
SELECT *
FROM (
    SELECT city, COUNT(*) AS total
    FROM student
    GROUP BY city
) AS temp
WHERE total > 2;

----------------------------------------------------------------------
SUBQUERY WITH EXISTS
----------------------------------------------------------------------
SELECT name
FROM student s
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.student_id = s.id
);

→ Returns students who have orders

----------------------------------------------------------------------
SUBQUERY WITH NOT EXISTS
----------------------------------------------------------------------
→ Opposite of EXISTS

----------------------------------------------------------------------
OPERATORS USED WITH SUBQUERY
----------------------------------------------------------------------
=        → single value
IN       → multiple values
ANY      → compare with any value
ALL      → compare with all values
EXISTS   → checks existence

----------------------------------------------------------------------
EDGE CASES
----------------------------------------------------------------------

1) SUBQUERY RETURNS NULL
----------------------------------------------------------------------
→ Comparison may return NULL (no match)

----------------------------------------------------------------------
2) MULTIPLE VALUES WITH "="
----------------------------------------------------------------------
SELECT * FROM student
WHERE age = (SELECT age FROM student);

→ Error (multiple rows returned)

----------------------------------------------------------------------
3) PERFORMANCE
----------------------------------------------------------------------
• Correlated subqueries are slow
• Use JOIN if possible

----------------------------------------------------------------------
4) EMPTY RESULT
----------------------------------------------------------------------
Subquery returns nothing → outer query may return no rows

----------------------------------------------------------------------
5) NESTED SUBQUERIES
----------------------------------------------------------------------
Subquery inside subquery → allowed but complex

----------------------------------------------------------------------
COMMON MISTAKES
----------------------------------------------------------------------

❌ Using = with multi-row subquery
❌ Not understanding execution order
❌ Overusing correlated subqueries

----------------------------------------------------------------------
EXECUTION FLOW
----------------------------------------------------------------------
1) Inner query executes
2) Result passed to outer query
3) Outer query runs

----------------------------------------------------------------------
NOTES
----------------------------------------------------------------------
1) Query inside query
2) Used for dynamic conditions
3) Can be in SELECT, WHERE, FROM
4) Powerful but can impact performance

RELATIONSHIPS IN MYSQL (COMPLETE NOTES)
======================================================================

DEFINITION:
Relationships define how tables are connected using keys
(usually PRIMARY KEY ↔ FOREIGN KEY)

----------------------------------------------------------------------
WHY RELATIONSHIPS ARE USED
----------------------------------------------------------------------
• Avoid data duplication
• Maintain data integrity
• Organize data logically
• Enable JOIN operations

----------------------------------------------------------------------
TYPES OF RELATIONSHIPS
----------------------------------------------------------------------

1) ONE-TO-ONE (1:1)
2) ONE-TO-MANY (1:N)
3) MANY-TO-MANY (M:N)

======================================================================
1) ONE-TO-ONE (1:1)
======================================================================

DEFINITION:
One row in Table A ↔ One row in Table B

----------------------------------------------------------------------
EXAMPLE
----------------------------------------------------------------------

users
-----------------------
id | name
1  | Ajay

profiles
-----------------------
id | user_id | address
1  | 1       | Delhi

----------------------------------------------------------------------
IMPLEMENTATION
----------------------------------------------------------------------

CREATE TABLE person(
    person_id INT PRIMARY KEY auto_increment,
    name VARCHAR(30);
)

CREATE TABLE profiles (
    id INT PRIMARY KEY,
    person_id INT UNIQUE,
    FOREIGN KEY (person_id) REFERENCES person(person_id)
);

→ UNIQUE ensures one-to-one

----------------------------------------------------------------------
USE CASE
----------------------------------------------------------------------
-security purpose
• User ↔ Profile
• Person ↔ Passport
3) EMPLOYEE ↔ ID CARD
4) VEHICLE ↔ REGISTRATION

WHEN TO USE 1:1
----------------------------------------------------------------------
• Split large table into smaller ones
• Separate sensitive data (security)
• Optional data (not every row needs it)

======================================================================
2) ONE-TO-MANY (1:N)
======================================================================

DEFINITION:
One row in parent table ↔ Many rows in child table

----------------------------------------------------------------------
EXAMPLE
----------------------------------------------------------------------

customers
-----------------------
id | name
1  | Ajay

orders
-----------------------
id | customer_id
1  | 1
2  | 1

→ One customer, many orders

----------------------------------------------------------------------
IMPLEMENTATION
----------------------------------------------------------------------
CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);

----------------------------------------------------------------------
USE CASE
----------------------------------------------------------------------
• Customer → Orders
• Department → Employees

======================================================================
3) MANY-TO-MANY (M:N)
======================================================================

DEFINITION:
Many rows in A ↔ Many rows in B

→ Requires a JUNCTION (bridge) table

----------------------------------------------------------------------
EXAMPLE
----------------------------------------------------------------------

students
-----------------------
id | name
1  | Ajay
2  | Rahul

courses
-----------------------
id | course_name
1  | Java
2  | SQL

student_course (junction table)
------------------------------------------
student_id | course_id
1          | 1
1          | 2
2          | 1

→ One student can take many courses
→ One course can have many students

-======================================================================
MANY-TO-MANY EXAMPLE (ALL 3 TABLES)
======================================================================

1) STUDENTS TABLE
----------------------------------------------------------------------
CREATE TABLE students (
    id INT PRIMARY KEY,
    name VARCHAR(50)
);

----------------------------------------------------------------------
2) COURSES TABLE
----------------------------------------------------------------------
CREATE TABLE courses (
    id INT PRIMARY KEY,
    course_name VARCHAR(50)
);

----------------------------------------------------------------------
3) JUNCTION TABLE (student_course)
----------------------------------------------------------------------
CREATE TABLE student_course (
    student_id INT,
    course_id INT,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES students(id),
    FOREIGN KEY (course_id) REFERENCES courses(id)
);

----------------------------------------------------------------------
SAMPLE DATA
----------------------------------------------------------------------

INSERT INTO students VALUES
(1, 'Ajay'),
(2, 'Rahul');

INSERT INTO courses VALUES
(1, 'Java'),
(2, 'SQL');

INSERT INTO student_course VALUES
(1, 1),
(1, 2),
(2, 1);

----------------------------------------------------------------------
OUTPUT UNDERSTANDING
----------------------------------------------------------------------

student_course table:
student_id | course_id
----------------------
1          | 1   → Ajay → Java
1          | 2   → Ajay → SQL
2          | 1   → Rahul → Java

→ One student → many courses
→ One course → many students

----------------------------------------------------------------------
NOTES
----------------------------------------------------------------------
1) Junction table connects both tables
2) Composite PK avoids duplicate pairs
3) Core concept for many-to-many relation

======================================================================
USE CASE
----------------------------------------------------------------------
• Students ↔ Courses
• Products ↔ Orders

======================================================================
KEY CONCEPTS
----------------------------------------------------------------------

PRIMARY KEY:
→ Unique identifier in a table

FOREIGN KEY:
→ Reference to another table

JUNCTION TABLE:
→ Used in many-to-many relationship

----------------------------------------------------------------------
RELATIONSHIP SUMMARY
----------------------------------------------------------------------

Type            Description
------------------------------------------
1:1             One ↔ One
1:N             One ↔ Many
M:N             Many ↔ Many (via junction)

----------------------------------------------------------------------
COMMON MISTAKES
----------------------------------------------------------------------

❌ Not using foreign keys
❌ Missing UNIQUE in 1:1
❌ Trying M:N without junction table
❌ Mismatched data types

----------------------------------------------------------------------
NOTES
----------------------------------------------------------------------
1) Relationships are backbone of database design
2) Implemented using FOREIGN KEY
3) Used heavily with JOINs
4) Ensures data consistency

======================================================================
COMPOSITE KEY (COMPOSITE PRIMARY KEY) — COMPLETE NOTES
======================================================================

DEFINITION:
A Composite Key is a PRIMARY KEY made using two or more columns
to uniquely identify a row.

----------------------------------------------------------------------
WHY COMPOSITE KEY IS USED
----------------------------------------------------------------------
• When a single column is NOT enough to ensure uniqueness
• To represent combination-based uniqueness
• Common in junction tables (many-to-many)

----------------------------------------------------------------------
BASIC SYNTAX
----------------------------------------------------------------------
CREATE TABLE table_name (
    col1 datatype,
    col2 datatype,
    PRIMARY KEY (col1, col2)
);

----------------------------------------------------------------------
EXAMPLE
----------------------------------------------------------------------

CREATE TABLE student_course (
    student_id INT,
    course_id INT,
    PRIMARY KEY (student_id, course_id)
);

→ Combination must be unique

----------------------------------------------------------------------
VALID & INVALID DATA
----------------------------------------------------------------------

INSERT INTO student_course VALUES (1, 1); ✔
INSERT INTO student_course VALUES (1, 2); ✔
INSERT INTO student_course VALUES (1, 1); ❌

→ Duplicate combination not allowed

----------------------------------------------------------------------
IMPORTANT RULES
----------------------------------------------------------------------

1) Combination must be UNIQUE
2) Individual columns can have duplicates
3) Cannot contain NULL values (PRIMARY KEY rule)

----------------------------------------------------------------------
COMPOSITE FOREIGN KEY
----------------------------------------------------------------------
CREATE TABLE orders (
    order_id INT,
    product_id INT,
    FOREIGN KEY (order_id, product_id)
    REFERENCES another_table(order_id, product_id)
);

→ References composite key

----------------------------------------------------------------------
WHEN TO USE
----------------------------------------------------------------------
• Many-to-many relationship tables
• When uniqueness depends on multiple columns
• Avoid creating artificial IDs when not needed

----------------------------------------------------------------------
ADVANTAGES
----------------------------------------------------------------------

✔ Ensures real-world uniqueness
✔ Avoids extra ID column
✔ Better normalization

----------------------------------------------------------------------
DISADVANTAGES
----------------------------------------------------------------------

❌ More complex queries
❌ Harder joins
❌ Larger indexes

----------------------------------------------------------------------
COMMON MISTAKES
----------------------------------------------------------------------

❌ Expecting individual column to be unique
❌ Using NULL values
❌ Forgetting order of columns matters

----------------------------------------------------------------------
NOTES
----------------------------------------------------------------------
1) Composite key = multiple columns as PK
2) Used heavily in junction tables
3) Enforces combination uniqueness
4) Important for database design

======================================================================
IS COMPOSITE PRIMARY KEY MUST?
======================================================================

ANSWER:
❌ NOT mandatory  
✔ But required ONLY if you want combination uniqueness

----------------------------------------------------------------------
WHEN YOU MUST USE IT
----------------------------------------------------------------------
If:
• No single column can uniquely identify a row  
• Uniqueness depends on multiple columns  

Example:
(student_id, course_id)

→ Same student can take multiple courses  
→ Same course can have multiple students  
→ But combination must be unique ✔

----------------------------------------------------------------------
WHEN NOT REQUIRED
----------------------------------------------------------------------
If you already have a unique column:

CREATE TABLE student_course (
    id INT AUTO_INCREMENT PRIMARY KEY,
    student_id INT,
    course_id INT
);

→ Works without composite key ✔

----------------------------------------------------------------------
BUT PROBLEM HERE
----------------------------------------------------------------------
INSERT INTO student_course VALUES (1, 1, 1);
INSERT INTO student_course VALUES (2, 1, 1); ❌ duplicate relation allowed

→ No restriction on (student_id, course_id)

----------------------------------------------------------------------
BETTER APPROACH
----------------------------------------------------------------------
Option 1: Composite PK ✔
PRIMARY KEY (student_id, course_id)

Option 2: Keep id + add UNIQUE ✔
UNIQUE (student_id, course_id)

----------------------------------------------------------------------
FINAL RULE
----------------------------------------------------------------------
• Not mandatory to write composite key  
• But MUST if you want to prevent duplicate combinations  

----------------------------------------------------------------------
INTERVIEW LINE
----------------------------------------------------------------------
"Composite key is not mandatory, but used when uniqueness
depends on multiple columns."

DDL vs DML
======================================================================

DEFINITION
----------------------------------------------------------------------
DDL (Data Definition Language)
→ Used to define or modify database structure

DML (Data Manipulation Language)
→ Used to manipulate (insert/update/delete) data

======================================================================
DDL COMMANDS
======================================================================

COMMANDS:
----------------------------------------------------------------------
CREATE   → create table/database
ALTER    → modify table structure
DROP     → delete table/database
TRUNCATE → delete all rows (structure remains)

EXAMPLES:
----------------------------------------------------------------------
CREATE TABLE student (id INT, name VARCHAR(50));

ALTER TABLE student ADD age INT;

DROP TABLE student;

TRUNCATE TABLE student;

----------------------------------------------------------------------
FEATURES
----------------------------------------------------------------------
• Works on structure (tables,schema,views)
• Auto-commit (cannot rollback)
• Faster execution

======================================================================
DML COMMANDS
======================================================================
-used for data operations
-works on rows(records)
COMMANDS:
----------------------------------------------------------------------
INSERT → add data
UPDATE → modify data
DELETE → remove data

EXAMPLES:
----------------------------------------------------------------------
INSERT INTO student VALUES (1, 'Ajay');

UPDATE student SET name = 'Rahul' WHERE id = 1;

DELETE FROM student WHERE id = 1;

----------------------------------------------------------------------
FEATURES
----------------------------------------------------------------------
• Works on data (rows)
• Can be rolled back (if transaction used)
• Slower than DDL

======================================================================
KEY DIFFERENCES
======================================================================

Feature        DDL                     DML
---------------------------------------------------------------
Purpose        Structure              Data
Commands       CREATE, ALTER, DROP    INSERT, UPDATE, DELETE
Rollback       Not possible           Possible
Auto-commit    Yes                    No
Affects        Table structure        Table data

======================================================================
NOTES
----------------------------------------------------------------------
1) DDL → design database
2) DML → work with data
3) Both are fundamental SQL concepts
