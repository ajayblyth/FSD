==================== NORMALIZATION IN MYSQL ====================

Definition:
• Normalization is the process of organizing data to reduce redundancy
• Ensures data consistency and efficient storage

---------------------------------------------------------------
WHY NORMALIZATION?
---------------------------------------------------------------

• Avoid duplicate data
• Maintain data consistency
• Reduce data anomalies (insert/update/delete issues)
• Improve database structure
==================== WHY WE NEED NORMALIZATION ====================

Definition:
• Normalization is the process of organizing data in a database
• Goal: reduce redundancy and improve data integrity

---------------------------------------------------------------
KEY REASONS
---------------------------------------------------------------

1) Eliminate Redundancy
• Avoid storing duplicate data in multiple places
• Saves storage and prevents inconsistencies

2) Ensure Data Integrity
• Each fact is stored in only one place
• Updates, inserts, deletes happen consistently

3) Avoid Anomalies
• **Insertion anomaly:** Cannot add data without unrelated data  
• **Update anomaly:** Changing data in one place requires multiple updates  
• **Deletion anomaly:** Deleting a record may remove needed data

4) Efficient Queries
• Smaller, well-structured tables → faster queries
• Relationships are clearly defined

5) Easier Maintenance
• Clear table structure → simpler to maintain and extend

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"Normalization organizes data to reduce redundancy, prevent anomalies,
and maintain integrity in relational databases."
---------------------------------------------------------------

---------------------------------------------------------------
NORMAL FORMS
---------------------------------------------------------------

1) 1NF (First Normal Form)
• No repeating groups
• Atomic values (no multiple values in one column)

Example:
❌ phone = '123,456'
✔ separate rows or table

---------------------------------------------------------------
2) 2NF (Second Normal Form)
• Must be in 1NF
• No partial dependency (all columns depend on full PK)

---------------------------------------------------------------
3) 3NF (Third Normal Form)
• Must be in 2NF
• No transitive dependency (non-key depends only on PK)

Example:
❌ student(id, dept_id, dept_name)
✔ split into:
   student(id, dept_id)
   department(dept_id, dept_name)

---------------------------------------------------------------
KEY POINT
---------------------------------------------------------------

• Break large tables into smaller related tables
• Use PRIMARY KEY + FOREIGN KEY for relationships

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"Normalization reduces redundancy and improves data integrity
by organizing data into well-structured tables."
---------------------------------------------------------------
==================== DENORMALIZED DATASET ====================

Definition:
• Denormalization means combining tables to reduce joins
• Data is intentionally duplicated for faster reads

---------------------------------------------------------------
EXAMPLE
---------------------------------------------------------------

-- Normalized (separate tables)
student(id, name, dept_id)
department(dept_id, dept_name)

-- Denormalized (combined)
student(id, name, dept_id, dept_name)

---------------------------------------------------------------
WHY USE DENORMALIZATION?
---------------------------------------------------------------

• Faster query performance (fewer joins)
• Useful in reporting / analytics
• Simpler queries

---------------------------------------------------------------
DISADVANTAGES
---------------------------------------------------------------

• Data redundancy (duplicate data)
• Risk of inconsistency
• More storage used

---------------------------------------------------------------
WHEN TO USE
---------------------------------------------------------------

• Read-heavy systems
• Reporting dashboards
• Data warehousing

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"Denormalization improves read performance by reducing joins,
but increases redundancy and risk of inconsistency."
---------------------------------------------------------------

==================== NORMALIZATION (STARTING WITH 1NF) ====================

==================== 1NF (FIRST NORMAL FORM) ====================

RULES:
• No repeating groups
• Each column must have atomic (single) values
• Each row should be uniquely identifiable (PRIMARY KEY)

---------------------------------------------------------------
VIOLATION EXAMPLE ❌
---------------------------------------------------------------

student
id | name | phone
1  | Ajay | 123,456

• Multiple values in one column → violates atomicity
• No proper unique identification for each phone

---------------------------------------------------------------
CORRECT (1NF) ✔
---------------------------------------------------------------

student
id | name | phone
1  | Ajay | 123
1  | Ajay | 456

• Atomic values ✔
• BUT id is duplicated → not uniquely identifiable ❌

---------------------------------------------------------------
FINAL CORRECT DESIGN ✔✔
---------------------------------------------------------------

student
id | name
1  | Ajay

phone
student_id | phone
1          | 123
1          | 456

• Each row is uniquely identifiable (PK can be added)
• No repeating groups
• Proper structure achieved

---------------------------------------------------------------
KEY POINT
---------------------------------------------------------------

• 1NF = atomic values + unique rows
• Use separate tables + primary keys when needed
• One cell = one value
• No comma-separated or multi-valued fields

---------------------------------------------------------------
NEXT STEP
---------------------------------------------------------------

• After 1NF → move to 2NF (remove partial dependency)
---------------------------------------------------------------

==================== 2NF (SECOND NORMAL FORM) ====================

DEFINITION:
• 2NF removes partial dependency
• All non-key columns must depend on FULL primary key

---------------------------------------------------------------
PREREQUISITE
---------------------------------------------------------------

• Must be in 1NF

---------------------------------------------------------------
RULES OF 2NF
---------------------------------------------------------------

• Table must be in 1NF
• No partial dependency allowed
• Every non-key attribute must depend on COMPLETE primary key


---------------------------------------------------------------
KEY CONCEPTS
---------------------------------------------------------------

1) Candidate Key
• Column(s) that can uniquely identify a row
• Multiple candidate keys possible
• One is selected as PRIMARY KEY

Example:
student(id, email)
→ Both id and email are candidate keys

---------------------------------------------------------------

2) Composite Key
• Primary key made of multiple columns

Example:
PRIMARY KEY (student_id, course_id)

---------------------------------------------------------------

3) Partial Dependency ❌
• Non-key column depends on ONLY part of composite key
• Not on full primary key → violates 2NF

---------------------------------------------------------------
VIOLATION EXAMPLE ❌
---------------------------------------------------------------

enrollments
student_id | course_id | student_name | course_name

PRIMARY KEY (student_id, course_id)

• student_name → depends only on student_id ❌
• course_name → depends only on course_id ❌
→ Partial dependency exists

---------------------------------------------------------------
CORRECT (2NF) ✔
---------------------------------------------------------------

student
student_id | student_name

course
course_id | course_name

enrollments
student_id | course_id

• No partial dependency
• Proper separation of data

---------------------------------------------------------------
CORRECT UNDERSTANDING (IMPORTANT)
---------------------------------------------------------------

• If non-key depends on FULL composite key → OK ✅
• If non-key depends on PART of composite key → NOT OK ❌

---------------------------------------------------------------
EXAMPLE
---------------------------------------------------------------

PRIMARY KEY (student_id, course_id)

Case 1: ✅ VALID
grade → depends on (student_id + course_id)
→ keep in same table

Case 2: ❌ INVALID
student_name → depends only on student_id
→ move to separate table

---------------------------------------------------------------
KEY POINTS
---------------------------------------------------------------

• 2NF applies only when composite key exists
• Split ONLY when partial dependency exists
• No split needed if dependency is on full key

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"2NF removes partial dependency by ensuring all non-key
attributes depend on the complete primary key."
---------------------------------------------------------------==================== RELATIONSHIP TABLE IN 2NF ====================

Definition:
• A relationship table (junction table) resolves many-to-many relationships
• Typically contains only foreign keys referencing related tables
• Can also have non-key attributes that depend on the FULL composite key

---------------------------------------------------------------
EXAMPLE
---------------------------------------------------------------

STUDENT
student_id | student_name

COURSE
course_id | course_name

ENROLLMENTS (relationship table)
student_id | course_id | grade

• PRIMARY KEY (student_id, course_id)
• grade depends on FULL composite key → 2NF satisfied

---------------------------------------------------------------
KEY POINTS
---------------------------------------------------------------

• Non-key attributes in relationship table must depend on the full composite key
• Avoid storing student_name or course_name here → partial dependency violation
• Junction table keeps normalized many-to-many relationships clean

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"Relationship tables in 2NF store only foreign keys and attributes
dependent on the full composite key, avoiding partial dependency."
---------------------------------------------------------------

==================== IS RELATIONSHIP TABLE MANDATORY FOR 2NF? ====================

• No, a relationship (junction) table is **not mandatory** for 2NF.  
• 2NF is about **removing partial dependency** on composite keys.  
• A relationship table is used **only when handling many-to-many relationships**.  
• If your table has a single-column primary key, 2NF is automatically satisfied; no relationship table is needed.  

---------------------------------------------------------------
KEY POINT
---------------------------------------------------------------

• 2NF focuses on dependency rules, not on creating junction tables.  
• Junction tables are a **design choice** to maintain normalization for many-to-many relationships.
---------------------------------------------------------------
==================== 3NF (THIRD NORMAL FORM) ====================

DEFINITION:
• 3NF removes transitive dependency
• Every non-key attribute must depend **only** on the primary key
• No non-key attribute should depend on another non-key attribute

---------------------------------------------------------------
PREREQUISITE
---------------------------------------------------------------

• Table must be in 2NF
• No partial dependency
• Must have a defined primary key

---------------------------------------------------------------
KEY CONCEPTS
---------------------------------------------------------------

1) Transitive Dependency ❌
• Non-key attribute depends on another non-key attribute
• Example:
  student_id → dept_id → dept_name
• dept_name depends on dept_id (non-key) → violates 3NF

2) Candidate Key
• Column(s) that uniquely identify a row
• Non-key attributes must depend on primary key only

---------------------------------------------------------------
VIOLATION EXAMPLE ❌
---------------------------------------------------------------

student
student_id | student_name | dept_id | dept_name

PRIMARY KEY (student_id)

• dept_name depends on dept_id (non-key) → transitive dependency  
• Violates 3NF

---------------------------------------------------------------
CORRECT (3NF) ✔
---------------------------------------------------------------

student
student_id | student_name | dept_id

department
dept_id | dept_name

• Non-key attributes depend only on primary key
• No transitive dependency

---------------------------------------------------------------
KEY POINTS
---------------------------------------------------------------

• 3NF eliminates indirect dependencies  
• Keeps data consistent and reduces redundancy  
• Improves update, insert, delete operations

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"3NF ensures that all non-key attributes depend only on the primary key,
removing transitive dependencies for better data integrity."

==================== TCL (TRANSACTION CONTROL LANGUAGE) ====================

Definition:
• TCL commands manage **transactions** in a database
• Ensure that a group of SQL statements execute **together** or **not at all**

---------------------------------------------------------------
STARTING A TRANSACTION
---------------------------------------------------------------

• BEGIN / START TRANSACTION  
  - Explicitly starts a transaction  
  - Groups multiple DML statements into a single unit  
  - Ensures atomicity: all succeed or all fail  

Example:
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;    -- permanently save changes
-- OR
ROLLBACK;  -- undo all changes

---------------------------------------------------------------
KEY COMMANDS
---------------------------------------------------------------

1) COMMIT
• Saves all changes made in the current transaction
• Makes them permanent

2) ROLLBACK
• Reverts all changes made in the current transaction
• Useful if an error occurs

3) SAVEPOINT
• Sets a point within a transaction to which you can rollback partially
• Useful for large transactions

Example:
SAVEPOINT sp1;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
ROLLBACK TO sp1;   -- undoes only after SAVEPOINT

---------------------------------------------------------------
KEY POINTS
---------------------------------------------------------------

• Transactions ensure **atomicity**: all or nothing  
• TCL commands only affect **DML operations** (INSERT, UPDATE, DELETE)  
• COMMIT = permanent, ROLLBACK = undo, SAVEPOINT = partial rollback  
• BEGIN ensures a clear start of transaction, especially for multiple statements

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"TCL commands control transactions to maintain data integrity
and consistency in relational databases. BEGIN marks the start
of a transaction before COMMIT or ROLLBACK."
---------------------------------------------------------------
==================== ACID PROPERTIES (DATABASE) ====================

Definition:
• ACID ensures reliable and consistent transactions in a database

---------------------------------------------------------------
A → ATOMICITY
---------------------------------------------------------------

• "All or Nothing"
• If one operation fails → entire transaction fails

Example:
Transfer money:
Debit success + Credit fail → ROLLBACK

---------------------------------------------------------------
C → CONSISTENCY
---------------------------------------------------------------

• Database must remain in valid state
• Rules, constraints must always be satisfied

Example:
Balance cannot be negative (if rule exists)

---------------------------------------------------------------
I → ISOLATION
---------------------------------------------------------------

• Transactions run independently
• One transaction should not affect another

Example:
Two users updating same data → handled safely

---------------------------------------------------------------
D → DURABILITY
---------------------------------------------------------------

• Once COMMIT is done → data is permanent
• Survives crash or power failure

---------------------------------------------------------------
KEY POINT
---------------------------------------------------------------

• ACID ensures data integrity + reliability in transactions

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"ACID properties guarantee reliable, consistent, and safe
transaction execution in a database system."
