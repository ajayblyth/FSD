git revert (Notes)
==============================

git revert
----------
- Used to undo a specific commit safely.
- Works only on committed changes.
- Does NOT delete history.
- Creates a NEW commit that reverses an old commit.

Where it works:
- Working Directory ❌
- Staging Area ❌
- Committed (Local Repo) ✅
- Pushed (Remote Repo) ✅

Syntax:
git revert <commit-hash>

What happens:
- Original commit stays in history
- New commit cancels the effect of that commit

Why use revert:
- Safe for shared repositories
- No history rewrite
- Best option after code is pushed

Example:
git log --oneline
git revert a1b2c3d

Key difference:
revert → undo via new commit (safe)
reset  → removes commit/history (unsafe for shared repos)

==============================
git revert --continue
==============================

git revert --continue
---------------------
- Used when a revert causes conflicts.
- Tells Git that conflicts are resolved.
- Completes the revert process.

When it is needed:
- During git revert, if files have merge conflicts
- Git pauses and asks you to fix conflicts manually

Steps:
1. git revert <commit-hash>
2. Fix conflicts in files
3. git add <fixed-files>
4. git revert --continue

Notes:
- A new revert commit is created after --continue
- If you want to cancel instead, use: git revert --abort

==============================
git reset (Notes)
==============================

git reset
---------
- Used to move HEAD to a previous commit.
- Can undo commits and/or unstage changes.
- Rewrites history (use carefully).

Areas it can affect:
- Working Directory ✅
- Staging Area ✅
- Local Commits ✅
- Remote Repo ❌ (until pushed)

Types of reset:

1. --soft
   git reset --soft <commit>
   git reset --soft Head~1
   - Moves HEAD only
   - Keeps changes in staging area
   - Commits removed, code safe

2. --mixed (default)
   git reset <commit>
   - Moves HEAD + unstages files
   - Changes remain in working directory

3. --hard
   git reset --hard <commit>
   or    git reset --hard Head~1

   - Removes commits + staged + working changes
   - Code is permanently deleted
   -risky

When to use:
- Before pushing to remote
- For local cleanup only

Warning:
- Do NOT use reset on pushed commits in shared repos

Purpose of Git History / Debugging
----------------------------------
- To understand what changed in the code and why
- To track bugs by checking past commits
- To know who made a change and when
- To recover lost commits or mistakes
- Helpful during debugging, reviews, and audits

----------------------------------------
git log
----------------------------------------
Purpose:
- Shows complete commit history in detail

Shows:
- commit hash
- author
- date
- commit message

Syntax:
git log

Use when:
- You want full history with details

----------------------------------------
git log --oneline
----------------------------------------
Purpose:
- Compact and readable commit history

Shows:
- short commit hash
- commit message (one line)

Syntax:
git log --oneline

Use when:
- You want quick overview of commits

----------------------------------------
git log <file_name>
----------------------------------------
Purpose:
- Shows history of a specific file only

Shows:
- commits where that file was changed

Syntax:
git log index.html

Use when:
- Debugging issues related to one file

----------------------------------------
git log --oneline --graph
----------------------------------------
Purpose:
- Visualizes branch and merge history

Shows:
- commit timeline
- branching and merging structure

Syntax:
git log --oneline --graph --all

Use when:
- Understanding branch flow

----------------------------------------
git show <commit-id>
----------------------------------------
Purpose:
- Shows details of a specific commit

Shows:
- code changes (diff)
- commit message
- author info

Syntax:
git show a1b2c3d

Use when:
- Inspecting what exactly changed in a commit

----------------------------------------
git blame <file_name>
----------------------------------------
Purpose:
- Shows who changed each line of a file

Shows:
- author name
- commit id
- date per line

Syntax:
git blame index.html

Use when:
- Finding responsibility for a bug
- Understanding why code was written

----------------------------------------
git reflog
----------------------------------------
Purpose:
- Tracks all HEAD movements (even deleted commits)

Shows:
- commits after reset, checkout, rebase
- recovery history

Syntax:
git reflog

Use when:
- Recovering lost commits
- Undoing reset or mistakes

Key Point:
- reflog is local only
- Entries expire over time

----------------------------------------
Quick Comparison
----------------------------------------

git log    -> official commit history
git reflog -> internal safety log

----------------------------------------
Debugging Tip
----------------------------------------
- Bug appeared today?
- Check git log
- Inspect commit with git show
- Find author using git blame
- Recover mistakes using git reflog

==============================
Pull Request (PR) – Production Level Explanation
==============================

What is a Pull Request?
----------------------
- A Pull Request (PR) is a request to merge your code changes
  from one branch into another (usually feature → main).
- It is NOT an automatic merge.
- It starts a review and approval process.

Why Pull Requests are used
--------------------------
- To review code before it goes to production
- To maintain code quality and standards
- To avoid breaking the main/production branch
- To discuss changes, improvements, and bugs

Typical Real-World Workflow
---------------------------
1. Developer creates a feature branch
   -> git branch feature-login
   -> git checkout feature-login

2. Developer writes code and commits changes
   -> git add .
   -> git commit -m "Add login feature"

3. Developer pushes branch to GitHub
   -> git push origin feature-login

4. Developer creates Pull Request on GitHub
   - Source: feature-login
   - Target: main (or develop)

--------------------------------
What Happens After PR is Created
--------------------------------
- PR is visible to the team
- Code review starts
- CI checks may run automatically
  (tests, linting, build checks)

--------------------------------
Who Reviews the Pull Request?
--------------------------------
- Senior Developer
- Tech Lead
- Team Lead
- Maintainer / Repo Owner

(Depends on company rules)

--------------------------------
What Reviewers Check
--------------------------------
- Code correctness
- Logic and edge cases
- Coding standards
- Performance impact
- Security issues
- Test coverage
- Naming conventions
- Side effects on existing code

--------------------------------
Possible PR Outcomes
--------------------------------
1. Approved
   - Code is good
   - Ready to merge

2. Changes Requested
   - Reviewer adds comments
   - Developer updates code
   - Pushes new commits
   - PR updates automatically

3. Rejected / Closed
   - Feature not needed
   - Wrong approach
   - Business decision

--------------------------------
Who Merges the Pull Request?
--------------------------------
- Usually NOT the developer who created it
- Merged by:
  -> Tech Lead
  -> Senior Developer
  -> Maintainer
  -> Release Manager

(Prevents self-approval in production)

--------------------------------
Merge Methods (Production)
--------------------------------
1. Merge Commit
   - Keeps full branch history
   - Common in enterprise projects

2. Squash and Merge
   - Combines all commits into one
   - Clean history
   - Very common in production

3. Rebase and Merge
   - Linear history
   - Used by experienced teams

--------------------------------
After Merge
--------------------------------
- Code becomes part of target branch
- Feature branch is usually deleted
- Deployment pipeline may trigger
- Code goes to staging / production

--------------------------------
Pull Request vs Direct Push
--------------------------------
Direct Push:
- Risky
- No review
- Not allowed in production

Pull Request:
- Reviewed
- Tested
- Controlled
- Safe for production

--------------------------------
Key Production Rule
--------------------------------
"Main branch is protected.
No direct commits.
Everything goes through Pull Requests."

--------------------------------
One-Line Summary
--------------------------------
A Pull Request is a controlled, reviewed,
and approved way to merge code into
production-ready branches.
