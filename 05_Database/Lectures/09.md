======================================================================
CASE IN MYSQL (COMPLETE STRUCTURED NOTES)
======================================================================

DEFINITION:
CASE is used for conditional logic (like IF-ELSE)

→ Returns value based on conditions

======================================================================
TYPES OF CASE
======================================================================

1) SIMPLE CASE
2) SEARCHED CASE

======================================================================
1) SIMPLE CASE
======================================================================

DEFINITION:
Compares a column/expression with fixed values (= only)

----------------------------------------------------------------------
SYNTAX
----------------------------------------------------------------------
CASE column
    WHEN value1 THEN result1
    WHEN value2 THEN result2
    ELSE result //default value
END
FROM table_name;

----------------------------------------------------------------------
EXAMPLE
----------------------------------------------------------------------
SELECT name, department,
CASE department
    WHEN 'IT' THEN 'Tech'
    WHEN 'HR' THEN 'Management'
    ELSE 'Other'
END AS dept_type
FROM employees;

----------------------------------------------------------------------
KEY POINT
----------------------------------------------------------------------
• Uses only equality (=) comparison  
• Cannot use >, <, BETWEEN  

======================================================================
2) SEARCHED CASE
======================================================================

DEFINITION:
Uses full conditions (>, <, =, BETWEEN, etc)

----------------------------------------------------------------------
SYNTAX
----------------------------------------------------------------------
CASE
    WHEN condition1 THEN result1
    WHEN condition2 THEN result2
    ELSE result
END

----------------------------------------------------------------------
EXAMPLE
----------------------------------------------------------------------
SELECT name, salary,
CASE
    WHEN salary > 5000 THEN 'High'
    WHEN salary BETWEEN 3000  AND 2500 THEN 'Medium'
    ELSE 'Low'
END AS salary_level
FROM employees;

----------------------------------------------------------------------
KEY POINT
----------------------------------------------------------------------
• Supports all conditions  
• More flexible and commonly used  

======================================================================
COMMON USAGE
======================================================================

1) IN SELECT (ADD NEW COLUMN)
----------------------------------------------------------------------
SELECT name, age,
CASE
    WHEN age >= 18 THEN 'Adult'
    ELSE 'Minor'
END AS status
FROM student;

----------------------------------------------------------------------
2) MULTIPLE CONDITIONS
----------------------------------------------------------------------
SELECT name, marks,
CASE
    WHEN marks >= 90 THEN 'A'
    WHEN marks >= 75 THEN 'B'
    WHEN marks >= 50 THEN 'C'
    ELSE 'Fail'
END AS grade
FROM student;

----------------------------------------------------------------------
3) WITH ORDER BY (CUSTOM SORT)
----------------------------------------------------------------------
EXAMPLE 1:
SELECT name, country
FOM employees
ORDER BY
CASE country
WHEN 'india' then 1
WHEN 'us' then 2
ELSE 3
END;

EXAMPLE 2:
SELECT name, department
FROM employees
ORDER BY
CASE
    WHEN department = 'IT' THEN 1
    WHEN department = 'HR' THEN 2
    ELSE 3
END;

----------------------------------------------------------------------
4) WITH UPDATE
----------------------------------------------------------------------
UPDATE employees
SET salary =
CASE
    WHEN performance_rating = 5 THEN salary + 10000
    WHEN performance_rating = 4 THEN salary + 5000
    WHEN performance_rating = 3 THEN salary + 2000
    ELSE salary
END;

----------------------------------------------------------------------
5) WITH AGGREGATE (CONDITIONAL COUNT)
----------------------------------------------------------------------
SELECT 
    SUM(CASE WHEN salary > 5000 THEN 1 ELSE 0 END) AS high_salary_count
FROM employees;

//using sum in recommended as it doesnt ignore null values.

5.1) USING COUNT WITH CASE, use it when you want custom columns

SELECT 
    COUNT(CASE WHEN salary > 5000 THEN 1 END) AS high_salary_count,
    COUNT(CASE WHEN salary BETWEEN 3000 AND 5000 THEN 1 END) AS medium_salary_count,
    COUNT(CASE WHEN salary < 3000 THEN 1 END) AS low_salary_count
FROM employees;

→ Counts only non-NULL values

HOW IT WORKS
----------------------------------------------------------------------
CASE returns:
• 1 → counted ✔  
• NULL → ignored  
----------------------------------------------------------------------
6) WITH WHERE (NOT RECOMMENDED)
----------------------------------------------------------------------
SELECT name
FROM student
WHERE
CASE
    WHEN age > 18 THEN 1
    ELSE 0
END = 1;

✔ Better:
SELECT name FROM student WHERE age > 18;

-----------------------------------------
7)GROUP BY
-----------------------------------------

EXAMPLE 1: COUNT EMPLOYEES BY SALARY CATEGORY
---------------------------------------------------------
SELECT 
    CASE
        WHEN salary >= 100000 THEN 'High'
        WHEN salary >= 50000 THEN 'Medium'
        ELSE 'Low'
    END AS salary_category,
    COUNT(*) AS total_employees
FROM employees
GROUP BY salary_category;


OUTPUT (Sample)
---------------------------------------------------------
salary_category | total_employees
--------------------------------
High            | 3
Medium          | 5
Low             | 2


=========================================================
EXAMPLE 2: TOTAL SALARY PER CATEGORY
=========================================================
SELECT 
    CASE
        WHEN salary >= 100000 THEN 'High'
        WHEN salary >= 50000 THEN 'Medium'
        ELSE 'Low'
    END AS salary_category,
    SUM(salary) AS total_salary
FROM employees
GROUP BY salary_category;


OUTPUT (Sample)
---------------------------------------------------------
salary_category | total_salary
--------------------------------
High            | 300000
Medium          | 250000
Low             | 80000


=========================================================
KEY POINTS
=========================================================
• CASE creates custom groups (High / Medium / Low)
• GROUP BY groups data based on CASE result
• Then we apply aggregate functions like COUNT, SUM

======================================================================
EDGE CASES
----------------------------------------------------------------------

1) NO MATCH
----------------------------------------------------------------------
→ Returns NULL (if ELSE not provided)

2) FIRST MATCH EXECUTES
----------------------------------------------------------------------
→ Stops after first true condition

3) NULL HANDLING
----------------------------------------------------------------------
CASE
    WHEN column IS NULL THEN 'Missing'
END

======================================================================
COMMON MISTAKES
----------------------------------------------------------------------

❌ Forgetting END  
❌ Wrong condition order  
❌ Using CASE instead of simple WHERE  

======================================================================
DIFFERENCE SUMMARY
----------------------------------------------------------------------

SIMPLE CASE        SEARCHED CASE
-----------------------------------------------
= comparison       Any condition
Fixed values       Flexible logic
Less powerful      More powerful

======================================================================
FINAL RULE
----------------------------------------------------------------------
• Simple CASE → exact value matching  
• Searched CASE → condition-based logic  

======================================================================
NOTES
----------------------------------------------------------------------
1) CASE = IF-ELSE in SQL  
2) Used in SELECT, ORDER BY, UPDATE  
3) Very powerful for data transformation  

======================================================================
======================================================================
CROSS QUESTIONS ON CASE (INTERVIEW / STUDENT DOUBTS)
======================================================================

1) What happens if no condition matches?
----------------------------------------------------------------------
→ Returns NULL (if ELSE not given)

----------------------------------------------------------------------
2) Does CASE execute all conditions?
----------------------------------------------------------------------
→ No, stops at FIRST matching condition

----------------------------------------------------------------------
3) Can we use >, < in simple CASE?
----------------------------------------------------------------------
→ No ❌ only "=" allowed  
→ Use searched CASE ✔

----------------------------------------------------------------------
4) Can CASE be used in WHERE?
----------------------------------------------------------------------
→ Yes, but not recommended  
→ Better to use direct conditions

----------------------------------------------------------------------
5) Can CASE return different data types?
----------------------------------------------------------------------
→ Should be same type (else implicit conversion happens)

----------------------------------------------------------------------
6) What if column value is NULL?
----------------------------------------------------------------------
→ Use:
WHEN column IS NULL THEN ...

----------------------------------------------------------------------
7) Which is better: CASE or IF()?
----------------------------------------------------------------------
→ CASE is standard SQL ✔  
→ IF() is MySQL-specific

----------------------------------------------------------------------
8) Can we use CASE with GROUP BY?
----------------------------------------------------------------------
→ Yes ✔ often used for grouping categories

Example:
SELECT 
CASE WHEN salary > 5000 THEN 'High' ELSE 'Low' END,
COUNT(*)
FROM employees
GROUP BY 1;

----------------------------------------------------------------------
9) Can CASE be nested?
----------------------------------------------------------------------
→ Yes ✔ (but avoid too much nesting)

----------------------------------------------------------------------
10) Is ELSE mandatory?
----------------------------------------------------------------------
→ No ❌ but recommended ✔

----------------------------------------------------------------------
11) Performance impact?
----------------------------------------------------------------------
→ Minimal, but complex CASE can slow queries

----------------------------------------------------------------------
12) Difference between CASE and WHERE?
----------------------------------------------------------------------
→ WHERE filters rows  
→ CASE transforms values

====================================
CTE (COMMON TABLE EXPRESSION) 
====================================

DEFINITION
----------------------------------------------------------------------
• A CTE is a temporary result set
• Defined using WITH keyword

• Used to simplify complex queries
• Exists only for that query (not stored permanently)

SYNTAX
----------------------------------------------------------------------
WITH cte_name AS (
    SELECT column1, column2
    FROM table_name
    WHERE condition
)
SELECT * FROM cte_name;


======================================================================
EXAMPLE 1: SIMPLE CTE
======================================================================
WITH high_salary AS (
    SELECT name, salary
    FROM employees
    WHERE salary > 50000
)
SELECT * FROM high_salary;


OUTPUT (Sample)
----------------------------------------------------------------------
name        | salary
--------------------
Ajay        | 70000
Rahul       | 80000


======================================================================
EXAMPLE 2: CTE WITH GROUP BY
======================================================================
WITH dept_salary AS (
    SELECT department, SUM(salary) AS total_salary
    FROM employees
    GROUP BY department
)
SELECT * FROM dept_salary;


OUTPUT (Sample)
----------------------------------------------------------------------
department  | total_salary
--------------------------
IT          | 200000
HR          | 120000

example2:

WITH high_salary_emps AS (
SELECT id, name, age, salary, department
FROM employees
WHERE salary > 5000),
department_salaries AS (
SELECT department, SUM(salary) AS total
FROM high_salary_emps GROUP BY department)
SELECT * FROM department_salaries;

======================================================================
EXAMPLE 3: CTE + JOIN
======================================================================
WITH dept_avg AS (
    SELECT department, AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department
)
SELECT e.name, e.salary, d.avg_salary
FROM employees e
JOIN dept_avg d
ON e.department = d.department;


======================================================================
KEY POINTS
======================================================================
• Improves readability of complex queries
• Can replace subqueries
• Can be reused multiple times in same query
• Not stored permanently (temporary)

======================================================================
==================== VIEW vs CTE ====================

DEFINITION
----------------------------------------------------------------------
VIEW:
• A virtual table stored in database
• Created using CREATE VIEW
• Can be reused anytime

CTE (Common Table Expression):
• Temporary result set
• Defined using WITH
• Exists only during query execution


SYNTAX
----------------------------------------------------------------------
VIEW:
CREATE VIEW view_name AS
SELECT column1, column2
FROM table_name;

CTE:
WITH cte_name AS (
    SELECT column1, column2
    FROM table_name
)
SELECT * FROM cte_name;


STORAGE
----------------------------------------------------------------------
VIEW:
• Stored in database

CTE:
• Not stored (temporary)


REUSABILITY
----------------------------------------------------------------------
VIEW:
• Can be reused in multiple queries

CTE:
• Used only once in that query


PERFORMANCE
----------------------------------------------------------------------
VIEW:
• Can improve performance (if indexed / materialized)

CTE:
• Mainly for readability, not performance


COMPLEX QUERIES
----------------------------------------------------------------------
VIEW:
• Good for repeated logic

CTE:
• Best for breaking complex queries into steps


EXAMPLE
----------------------------------------------------------------------
VIEW:
CREATE VIEW high_salary AS
SELECT name, salary
FROM employees
WHERE salary > 50000;

SELECT * FROM high_salary;


CTE:
WITH high_salary AS (
    SELECT name, salary
    FROM employees
    WHERE salary > 50000
)
SELECT * FROM high_salary;


KEY DIFFERENCE (1 LINE)
----------------------------------------------------------------------
• VIEW = permanent reusable query
• CTE  = temporary one-time query

==================== WINDOW FUNCTIONS (FINAL ORGANISED NOTES) ====================

DEFINITION
----------------------------------------------------------------------
• Perform calculations across a set of rows (window)
• Do NOT collapse rows (unlike GROUP BY)
• Each row keeps its identity


==================== AGGREGATE vs WINDOW ====================
----------------------------------------------------------------------
• Aggregate → collapses rows into ONE result per group
• Window    → keeps all rows + adds calculated column

• Aggregate uses GROUP BY
• Window uses OVER()

• Aggregate loses row-level detail
• Window keeps row-level detail


==================== SYNTAX ====================
----------------------------------------------------------------------
function_name(column) OVER (
    PARTITION BY column
    ORDER BY column
)


==================== IMPORTANT TERMS ====================
----------------------------------------------------------------------
• OVER()        → defines window
• PARTITION BY  → divides data into groups (keeps rows)
• ORDER BY      → defines order inside partition


==================== WINDOW FRAME (ADVANCED) ====================
----------------------------------------------------------------------
ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
→ considers full partition (start to end)


==================== EXAMPLE: MAX SALARY PER DEPT ====================
----------------------------------------------------------------------
SELECT *,
       MAX(salary) OVER (
           PARTITION BY department
           ORDER BY salary
           ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
       ) AS max_salary
FROM employees;

LOGIC:
• Partition by department
• Looks at ALL rows in that department
• Same max value shown in each row


======================================================================
EXAMPLE 1: ROW_NUMBER()
======================================================================
SELECT name, department, salary,
       ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS rank
FROM employees;

• Unique ranking (no duplicates)
• Highest salary → rank 1


---------------- IPL EXAMPLE ----------------
SELECT *,
       ROW_NUMBER() OVER (
           PARTITION BY match_id
           ORDER BY runs DESC
       ) AS rn
FROM ipl_matches;

• Resets numbering per match
• Highest runs → rn = 1


---------------- TOP SCORER USING CTE ----------------
WITH top_scorer AS (
    SELECT *,
           ROW_NUMBER() OVER (
               PARTITION BY match_id
               ORDER BY runs DESC
           ) AS rn
    FROM ipl_matches
)
SELECT * FROM top_scorer
WHERE rn = 1;

NOTE:
• ROW_NUMBER() → different rank for same values
• Use RANK() if same rank needed


======================================================================
EXAMPLE 2: RANK() vs DENSE_RANK()
======================================================================
SELECT name, salary,
       RANK() OVER (ORDER BY salary DESC) AS rnk,
       DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rnk
FROM employees;

RANK():        1, 2, 2, 4   (skips)
DENSE_RANK():  1, 2, 2, 3   (no skip)


======================================================================
EXAMPLE 3: RUNNING TOTAL
======================================================================
SELECT name, salary,
       SUM(salary) OVER (ORDER BY salary) AS running_total
FROM employees;


======================================================================
EXAMPLE 4: PARTITION + SUM
======================================================================
SELECT department, name, salary,
       SUM(salary) OVER (PARTITION BY department) AS dept_total
FROM employees;


======================================================================
EXAMPLE 5: LAG / LEAD (ROW COMPARISON)
======================================================================
SELECT date, sales,
       LAG(sales, 1) OVER (ORDER BY date) AS prev_day_sales,
       sales - LAG(sales, 1) OVER (ORDER BY date) AS change
FROM sales;

SELECT date, sales,
       LEAD(sales, 1) OVER (ORDER BY date) AS next_day_sales,
       LEAD(sales, 1) OVER (ORDER BY date) - sales AS change
FROM sales;

• LAG()  → previous row
• LEAD() → next row
• Used for trend / day-to-day analysis


======================================================================
EXAMPLE 6: NTILE() (BUCKETING)
======================================================================
SELECT name, salary,
       NTILE(4) OVER (ORDER BY salary DESC) AS bucket
FROM employees;

• Divides data into equal groups

--- MORE CASES ---
SELECT name, salary,
       NTILE(2) OVER (ORDER BY salary DESC) AS half
FROM employees;

• 1 → top 50%
• 2 → bottom 50%

SELECT department, name, salary,
       NTILE(3) OVER (PARTITION BY department ORDER BY salary DESC) AS bucket
FROM employees;

• Bucket inside each department

KEY IDEA
======================================================================
• Use when you need calculations WITHOUT losing rows

FINAL KEY POINTS
======================================================================
• GROUP BY → merges rows
• WINDOW → keeps rows

• Uses OVER() clause
• Used in analytics, reporting, interviews

• Types:
  - Ranking    → ROW_NUMBER, RANK, DENSE_RANK
  - Aggregate  → SUM, AVG OVER()
  - Value      → LAG, LEAD
  - Bucketing  → NTILE

================ COMMON DOUBTS IN WINDOW FUNCTIONS =================

1. WHY NOT USE GROUP BY?
---------------------------------------------------------
• GROUP BY → merges rows
• WINDOW → keeps rows
• Use WINDOW when you need row-level + aggregate together

2. DIFFERENCE BETWEEN PARTITION BY & GROUP BY?
---------------------------------------------------------
• GROUP BY → reduces rows
• PARTITION BY → divides data but keeps all rows

3. IS ORDER BY MANDATORY?
---------------------------------------------------------
• Not always
• Required for ranking, running total
• Optional for simple partition calculations

4. WHY OVER() IS REQUIRED?
---------------------------------------------------------
• Defines window (scope of calculation)
• Without OVER() → function becomes aggregate

5. CAN WE USE WHERE WITH WINDOW FUNCTION?
---------------------------------------------------------
• NO (directly)
• Use subquery or CTE

Example:
SELECT * FROM (
    SELECT name, salary,
           RANK() OVER (ORDER BY salary DESC) rnk
    FROM employees
) t
WHERE rnk <= 3;

6. PERFORMANCE?
---------------------------------------------------------
• Slightly heavy on large data
• But optimized in modern DBs

7. DIFFERENCE BETWEEN ROW_NUMBER, RANK, DENSE_RANK?
---------------------------------------------------------
• ROW_NUMBER → unique rank (1,2,3)
• RANK → skips duplicates (1,2,2,4)
• DENSE_RANK → no skip (1,2,2,3)
