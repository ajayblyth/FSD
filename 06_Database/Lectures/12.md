==================== INDEX IN MYSQL ====================

Definition:
• Index is a database object that improves query performance
• Works like a book index → helps find data faster

.

🧠 Simple idea
Without index → DB scans entire table (slow)
With index → DB uses shortcut (fast)
---------------------------------------------------------------
WHY USE INDEX
---------------------------------------------------------------

• Faster SELECT queries
• Quick searching, sorting, filtering
• Avoid full table scan

---------------------------------------------------------------
SYNTAX
---------------------------------------------------------------

-- Create index
CREATE INDEX idx_name
ON table_name(column_name);

---------------------------------------------------------------
EXAMPLE
---------------------------------------------------------------

CREATE INDEX idx_name
ON users(name);

-- Query becomes faster
SELECT * FROM users WHERE name = 'Ajay';

---------------------------------------------------------------
TYPES OF INDEX
---------------------------------------------------------------

1) PRIMARY KEY
• Automatically indexed
• Unique + NOT NULL

2) UNIQUE INDEX
• No duplicate values allowed

3) NORMAL INDEX
• Allows duplicates

4) COMPOSITE INDEX
• Index on multiple columns

Example:
CREATE INDEX idx_multi
ON users(name, city);

---------------------------------------------------------------
KEY POINTS
---------------------------------------------------------------

• Speeds up READ operations
• Slows down INSERT/UPDATE (index maintenance)
• Uses extra storage

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"Index improves query performance by allowing faster data
retrieval, similar to an index in a book."

==================== WHY CREATE INDEX IF PRIMARY KEY EXISTS ====================

TABLE:
---------------------------------------------------------------------
users(id PRIMARY KEY, name, email)

---------------------------------------------------------------------
CORE IDEA
---------------------------------------------------------------------
• Primary Key already creates an index ✔
• BUT it works only for that column (id)

---------------------------------------------------------------------
WORKING
---------------------------------------------------------------------

CASE 1: SEARCH USING PRIMARY KEY
---------------------------------------------------------------------
SELECT * FROM users WHERE id = 10;

✔ Fast (uses primary key index)


CASE 2: SEARCH USING OTHER COLUMN
---------------------------------------------------------------------
SELECT * FROM users WHERE email = 'a@gmail.com';

❌ Slow (full table scan)
• Primary key index on id is NOT used


---------------------------------------------------------------------
SOLUTION (CREATE EXTRA INDEX)
---------------------------------------------------------------------
CREATE INDEX idx_email ON users(email);


---------------------------------------------------------------------
AFTER INDEX
---------------------------------------------------------------------
SELECT * FROM users WHERE email = 'a@gmail.com';

✔ Fast (uses email index)


---------------------------------------------------------------------
KEY IDEA
---------------------------------------------------------------------
• Primary key index → only for id
• Extra index → for other searchable columns


---------------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------------
Primary key index works only on its column, so additional indexes
are created on frequently queried columns to avoid full table scans.
---------------------------------------------------------------------

📌 Index uses B-Tree (Balanced Tree)

👉 In most databases (like MySQL), indexes are stored as a B-Tree
==================== B-TREE & B+ TREE INDEXING ====================

---------------------------------------------------------------
1) B-TREE INDEX
---------------------------------------------------------------

Definition:
• Balanced tree data structure used for indexing
• Data stored in sorted order
• Nodes contain keys + data + child pointers

⚡ Why it’s fast
Tree is balanced
Each step removes half (or more) data
Very few comparisons
---------------------------------------------------------------
STRUCTURE
---------------------------------------------------------------

                [50]
              /      \
        [20,30]      [60,80]
        /    \        /    \
   [10]   [25]   [55]   [90]

• Data can exist at ANY level (root/intermediate/leaf)

---------------------------------------------------------------
HOW IT WORKS
---------------------------------------------------------------

• Start from root
• Compare value → move left/right
• May find data before reaching leaf

Example:
Search 30
→ 30 < 50 → go left
→ reach [20,30] → found

---------------------------------------------------------------
KEY POINT
---------------------------------------------------------------

• Search time → O(log n)
• Faster than linear search

---------------------------------------------------------------
2) B+ TREE INDEX (USED IN MYSQL) ⭐
---------------------------------------------------------------

Definition:
• Improved version of B-Tree
• Used in MySQL (InnoDB)

---------------------------------------------------------------
STRUCTURE
---------------------------------------------------------------

                [50]
              /      \
        [20,30]      [60,80]
        /    \        /    \
   [10,20] [25,30] [55,60] [80,90]

Leaf Nodes (Linked):
[10,20] ⇄ [25,30] ⇄ [55,60] ⇄ [80,90]

• Internal nodes → only keys
• Leaf nodes → actual data
• Leaf nodes connected sequentially

---------------------------------------------------------------
HOW IT WORKS
---------------------------------------------------------------

Search:
• Always traverse from root → leaf
• Data found ONLY at leaf level

Example:
Search 30
→ 30 < 50 → go left
→ reach [20,30]
→ go to leaf [25,30] → found

---------------------------------------------------------------
RANGE QUERY (ADVANTAGE)
---------------------------------------------------------------

Find values between 25 and 80:

• Start at first match → [25,30]
• Traverse via links:
  [25,30] → [55,60] → [80,90]

• No need to go back to root

---------------------------------------------------------------
WHY B+ TREE IS BETTER
---------------------------------------------------------------

• Faster range queries (linked leaves)
• More keys per node → smaller tree height
• Better disk I/O performance

---------------------------------------------------------------
DIFFERENCE
---------------------------------------------------------------

Feature        | B-TREE                  | B+ TREE
---------------|--------------------------|-------------------------
Data Storage   | In all nodes            | Only in leaf nodes
Search         | Can stop early          | Must reach leaf
Range Query    | Slower                  | Faster
Usage          | General                 | Databases (MySQL)

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"MySQL uses B+ Tree indexing because it provides faster search
and efficient range queries using linked leaf nodes."
---------------------------------------------------------------
==================== TIME COMPLEXITY (HIGH LEVEL) ====================

Definition:
• Time complexity measures how execution time grows with input size (n)
• Focus is on growth pattern, not exact time

---------------------------------------------------------------
COMMON TYPES
---------------------------------------------------------------

O(1)   → Constant time (no change with input size)  
O(log n) → Logarithmic (very fast growth, e.g., binary search)  
O(n)   → Linear (time increases with input size)  
O(n log n) → Efficient for sorting (merge sort, quicksort avg)  
O(n²)  → Quadratic (nested loops, slower for large data)

---------------------------------------------------------------
INTUITION
---------------------------------------------------------------

• Small n → difference not visible  
• Large n → huge impact on performance  

Example:
n = 1,000,000  
O(n)   → 1,000,000 steps  
O(log n) → ~20 steps  

---------------------------------------------------------------
WHY IMPORTANT
---------------------------------------------------------------

• Helps choose efficient algorithms  
• Predicts performance for large data  
• Critical for optimization (DB queries, indexing)

---------------------------------------------------------------
DB CONTEXT
---------------------------------------------------------------

• Index search (B+ Tree) → O(log n)  
• Full table scan → O(n)  

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"Time complexity shows how an algorithm scales with input size,
helping us choose efficient solutions."
---------------------------------------------------------------

==================== INDEX TYPES & CONCEPTS (MYSQL) ====================

---------------------------------------------------------------
DEFINITION
---------------------------------------------------------------

• Index = data structure that improves SELECT query performance
• Works like a book index → faster data lookup

---------------------------------------------------------------
TYPES OF INDEX
---------------------------------------------------------------

1) PRIMARY INDEX
• Created automatically on PRIMARY KEY
• Unique + NOT NULL
• Only one per table

---------------------------------------------------------------

2) UNIQUE INDEX
• Ensures all values are unique
• Allows NULL (once or multiple depending on DB)

CREATE UNIQUE INDEX idx_email
ON employees(email);

---------------------------------------------------------------

3) NORMAL INDEX
• Improves search performance
• Allows duplicate values

CREATE INDEX idx_salary
ON employees(salary);

---------------------------------------------------------------

4) COMPOSITE INDEX
• Index on multiple columns

CREATE INDEX idx_dept_salary
ON employees(department, age);

---------------------------------------------------------------
LEFTMOST PREFIX RULE ⭐
---------------------------------------------------------------

• Composite index works from LEFT to RIGHT

Index: (department, age)

Works:
✔ WHERE department = 'IT'
✔ WHERE department = 'IT' AND age > 25

Does NOT work efficiently:
❌ WHERE age > 25   (skips first column)

---------------------------------------------------------------

5) FULLTEXT INDEX
• Used for searching large text data
• Works with MATCH() AGAINST()

CREATE FULLTEXT INDEX idx_desc
ON table_name(description);

6) DROP INDEX 


Definition:
• Removes an existing index from a table
• Does NOT delete data, only removes index structure

DROP INDEX index_name
ON table_name;
---------------------------------------------------------------
EXAMPLE QUERIES (INDEX USAGE)
---------------------------------------------------------------

SELECT name, department 
FROM employees 
WHERE department = 'IT';

SELECT name, department, age 
FROM employees 
WHERE department = 'IT' AND age > 25;

---------------------------------------------------------------
DISADVANTAGES
---------------------------------------------------------------

• Slows down INSERT / UPDATE / DELETE  
  → because index also needs to be updated  

• Takes extra storage space  

• Too many indexes can degrade performance  

---------------------------------------------------------------
WHEN TO USE
---------------------------------------------------------------

• Frequent SELECT queries  
• WHERE, JOIN, ORDER BY columns  
• Large tables with filtering

---------------------------------------------------------------
KEY POINT
---------------------------------------------------------------

• Use indexes for READ-heavy systems  
• Avoid over-indexing  
==================== INDEX BEST PRACTICES & CASES ====================

---------------------------------------------------------------
IMPORTANT RULES
---------------------------------------------------------------

1) Avoid Functions on Indexed Columns ❌
• Index is NOT used if column is wrapped in function

❌ SELECT * FROM users WHERE UPPER(name) = 'AJAY';
✔ SELECT * FROM users WHERE name = 'Ajay';

---------------------------------------------------------------

2) Avoid Calculations on Indexed Columns ❌
❌ WHERE salary + 1000 > 5000
✔ WHERE salary > 4000

---------------------------------------------------------------

3) Use Index on JOIN Columns ✔
• Speeds up joins significantly

SELECT * 
FROM orders o
JOIN users u ON o.user_id = u.id;

→ Index on user_id and id helps

---------------------------------------------------------------

4) Use Index on WHERE / FILTER Columns ✔
• Most benefit when filtering large data

WHERE department = 'IT'

---------------------------------------------------------------

5) Use Index on ORDER BY / GROUP BY ✔
• Helps sorting and grouping

ORDER BY salary DESC

---------------------------------------------------------------

6) Composite Index Efficiency ⚠
• Benefit depends on usage

Example:
Index (A, B, C)

✔ WHERE A = ?
✔ WHERE A = ? AND B = ?
✔ WHERE A = ? AND B = ? AND C = ?

❌ WHERE B = ?  (skips A → no use)

---------------------------------------------------------------

7) Selectivity Matters ⭐
• High uniqueness → better index

✔ email (unique)
❌ gender (few values → low benefit)

---------------------------------------------------------------

8) Too Many Indexes ❌
• Slows down INSERT / UPDATE / DELETE
• Uses more storage

---------------------------------------------------------------

9) Small Tables ❌
• Index often not useful (full scan is faster)

---------------------------------------------------------------

10) Range Queries ⚠
• Index works, but stops further columns

WHERE department = 'IT' AND age > 25
→ age is range → next columns ignored

---------------------------------------------------------------
REALITY CHECK (COMMON MISCONCEPTION)
---------------------------------------------------------------

• “2 columns = 50%, 3 columns = 33% faster” ❌ WRONG

✔ Performance depends on:
  - Data size
  - Selectivity
  - Query pattern
  - Index usage

---------------------------------------------------------------
KEY POINT
---------------------------------------------------------------

• Index helps only when used correctly in query
• Wrong usage → index ignored

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"Indexes improve performance only when used on proper columns
without functions, with good selectivity, and aligned with query patterns."
---------------------------------------------------------------


---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"Indexes improve read performance but add overhead to write
operations, so they should be used wisely."

=====================================================================
CAP THEOREM (DISTRIBUTED TRADE-OFFS)
=====================================================================

Definition:
---------------------------------------------------------------------
• CAP theorem states that a distributed system can provide ONLY TWO out of:
  → Consistency (C)
  → Availability (A)
  → Partition Tolerance (P)

---------------------------------------------------------------------
DIAGRAM (TRADE-OFF TRIANGLE)
---------------------------------------------------------------------

              CONSISTENCY (C)
                   /\
                  /  \
                 /    \
                /      \
               /        \
              /          \
     (CP)    /            \    (CA)
            /              \
           /                \
          /                  \
         /____________________\
   PARTITION (P)         AVAILABILITY (A)

---------------------------------------------------------------------
PROPERTIES
---------------------------------------------------------------------

1) CONSISTENCY (C)
---------------------------------------------------------------------
 All users see SAME data at the same time
After update → everyone gets latest value

Example:
User1 updates value to 100 → User2 also sees 100 ✔

---------------------------------------------------------------------

2) AVAILABILITY (A)
---------------------------------------------------------------------
• System always responds (success/failure)
• No request is left unanswered

Example:
Server replies even if data is not latest ✔

---------------------------------------------------------------------

3) PARTITION TOLERANCE (P)
---------------------------------------------------------------------
• System continues working despite network failure
• Nodes can’t communicate but system still runs

---------------------------------------------------------------------
IMPORTANT RULE ⭐
---------------------------------------------------------------------

• In real distributed systems → P is mandatory
• So choice is between:
  → CP (Consistency + Partition tolerance)
  → AP (Availability + Partition tolerance)

---------------------------------------------------------------------
EXAMPLES
---------------------------------------------------------------------

CP Systems:
• Prioritizes correct data
• May delay or reject response
• Example: Banking systems

AP Systems:
• Always available, may return outdated  data
• Example: Social media feeds

CA Systems:
• Only possible when no network partition
• Not practical in real distributed systems
- possible or available on local system only

---------------------------------------------------------------------
KEY POINT
---------------------------------------------------------------------

• You cannot achieve all 3 simultaneously in distributed systems

---------------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------------

• CAP theorem says a distributed system can guarantee ONLY 2 out of 3:

C → Consistency
A → Availability
P → Partition Tolerance

---------------------------------------------------------------------
 CA (Consistency + Availability) ⚠
---------------------------------------------------------------------

Use when:
• No network partition (single node / local system)

Examples:
• Traditional RDBMS (single server)

Reality:
• Not practical in distributed systems
• Because network failures (P) always possible

---------------------------------------------------------------------
WHY CP OVER CA? ⭐
---------------------------------------------------------------------

• In distributed systems → partitions WILL happen
• So P cannot be ignored

→ Choosing CA means ignoring partition → unrealistic

✔ Hence real choice:
   CP vs AP (NOT CA)

---------------------------------------------------------------------
WHAT IS SYNCHRONOUS HERE?
---------------------------------------------------------------------

Synchronous (related to Consistency):

• All nodes must be updated BEFORE response is given
• System waits for confirmation from replicas

Example:
Write data → replicate to all nodes → then respond

✔ Ensures strong consistency  
❌ Slower (waiting time)

---------------------------------------------------------------------

Asynchronous (related to Availability):

• Response is given immediately
• Data replicated later

Example:
Write data → respond → update other nodes later

✔ Faster, always available  
❌ Temporary inconsistency

---------------------------------------------------------------------
REAL-WORLD ANALOGY
---------------------------------------------------------------------

Bank (CP):
• Better to reject transaction than show wrong balance

Instagram (AP):
• Better to show slightly old posts than fail to load

---------------------------------------------------------------------
KEY TRADE-OFF
---------------------------------------------------------------------

CP → Accuracy over Availability  
AP → Availability over Accuracy  

---------------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------------

"In distributed systems, partition tolerance is unavoidable,
so we choose between consistency (CP) and availability (AP)
based on whether correctness or uptime is more critical."
---------------------------------------------------------------------

==================== OLTP vs OLAP ====================

---------------------------------------------------------------
DEFINITION
---------------------------------------------------------------

OLTP (Online Transaction Processing)
• Handles day-to-day transactions  
• Focus on fast inserts/updates  

OLAP (Online Analytical Processing)
• Used for analysis and reporting  
• Focus on complex queries  

---------------------------------------------------------------
UNDERSTANDING
---------------------------------------------------------------

OLTP:
• System designed for handling transactions  
• Uses simple queries (INSERT, UPDATE, DELETE)

OLAP:
• System designed for analysis  
• Uses complex queries (SELECT, GROUP BY, JOIN)

---------------------------------------------------------------
COMPARISON
---------------------------------------------------------------

Feature        | OLTP                          | OLAP
---------------|-------------------------------|------------------------------
Purpose        | Transaction processing        | Data analysis / reporting
Users          | End users (apps)              | Analysts, managers
Queries        | Simple, frequent              | Complex, fewer
Operations     | INSERT, UPDATE, DELETE        | SELECT (aggregations)
Data Size      | Small per transaction         | Large historical data
Response Time  | Very fast (ms)                | Slower (sec/min)
Schema         | Normalized                    | Denormalized
Examples       | Banking, orders               | Data warehouse, BI

---------------------------------------------------------------
EXAMPLE
---------------------------------------------------------------

OLTP:
INSERT INTO orders VALUES (...);
UPDATE accounts SET balance = balance - 100;

OLAP:
SELECT department, AVG(salary)
FROM employees
GROUP BY department;

---------------------------------------------------------------
IMPORTANT NOTE ⭐
---------------------------------------------------------------

• Do NOT run analytical queries in OLTP systems ❌  
  → Can slow down transactions  

• OLTP → optimized for writes (transactions)  
• OLAP → optimized for reads (analysis)

---------------------------------------------------------------
KEY POINT
---------------------------------------------------------------

• OLTP = transactional system (real-time operations)  
• OLAP = analytical system (decision-making)

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"OLTP handles real-time transactions, while OLAP is used for
analyzing large volumes of data; they are system types, not just queries."
