======================================================================
MYSQL JOINS (COMPLETE DETAILED NOTES)
======================================================================

DEFINITION:
JOIN is used to combine rows from two or more tables
based on a related column (PRIMARY KEY ↔ FOREIGN KEY)

Note: join vs foreign key
"JOIN is used to fetch related data, while foreign key is used to enforce relationship integrity. 
JOIN works even without foreign key."
----------------------------------------------------------------------
WHY JOINS ARE USED
----------------------------------------------------------------------
• Fetch related data from multiple tables
• Avoid data duplication
• Build meaningful queries

----------------------------------------------------------------------
TABLES (FOR EXAMPLES)
----------------------------------------------------------------------

students
-----------------------
id | name
1  | Ajay
2  | Rahul
3  | Neha

orders
-----------------------
id | student_id
1  | 1
2  | 1
3  | 2

======================================================================
1) INNER JOIN
======================================================================

DEFINITION:
Returns only matching rows in BOTH tables

----------------------------------------------------------------------
SYNTAX
----------------------------------------------------------------------
SELECT columns
FROM table1
INNER JOIN table2
ON condition;

Note:if you only mention JOIN , my sql will take that as INNER JOIN as well by default.

----------------------------------------------------------------------
EXAMPLE
----------------------------------------------------------------------
SELECT s.name, o.id
FROM students s
INNER JOIN orders o
ON s.id = o.student_id;

→ OUTPUT:
Ajay   1
Ajay   2
Rahul  3

→ Neha not included (no matching order)

----------------------------------------------------------------------
REAL EXAMPLE WITH CONDITION
----------------------------------------------------------------------
SELECT customers.name, orders.order_date 
FROM customers
INNER JOIN orders
ON customers.customer_id = orders.customer_id
WHERE customers.name = "John";

----------------------------------------------------------------------
INNER JOIN WITH ALIAS
----------------------------------------------------------------------
SELECT c.name, o.order_date
FROM customers c
INNER JOIN orders o
ON c.customer_id = o.customer_id
WHERE c.name = "John";

======================================================================
INNER JOIN WITHOUT ON CONDITION
======================================================================

CASE 1: Standard INNER JOIN
----------------------------------------------------------------------
SELECT * 
FROM students s
INNER JOIN orders o;

→ ERROR ❌ (MySQL requires ON / USING)

----------------------------------------------------------------------
CASE 2: OLD STYLE (COMMA)
----------------------------------------------------------------------
SELECT * 
FROM students, orders;

→ Cartesian Product ✔

→ Every row of students × every row of orders

Example:
3 students × 3 orders = 9 rows

----------------------------------------------------------------------
IMPORTANT UNDERSTANDING
----------------------------------------------------------------------
• INNER JOIN MUST have ON condition  
• Without condition → use comma → gives Cartesian product  
• CROSS JOIN explicitly gives Cartesian product

----------------------------------------------------------------------
FINAL RULE
----------------------------------------------------------------------
No condition → Cartesian product  
Proper JOIN → meaningful data  

======================================================================
2) OUTER JOIN
======================================================================

DEFINITION:
OUTER JOIN returns matching rows + non-matching rows
(with NULLs for missing side)

----------------------------------------------------------------------
TYPES OF OUTER JOIN
----------------------------------------------------------------------
• LEFT JOIN
• RIGHT JOIN
• FULL JOIN (not direct in MySQL)

NOTE:
🔹 The table written before LEFT JOIN is the left table
👉 The table written after RIGHT JOIN is the right table
======================================================================
2.1) LEFT JOIN (LEFT OUTER JOIN)
======================================================================

DEFINITION:
Returns ALL rows from LEFT table + matching rows from RIGHT table
👉 If no match → NULL values from right table
EXAMPLE
----------------------------------------------------------------------
SELECT s.name, o.id
FROM students s
LEFT JOIN orders o
ON s.id = o.student_id;

→ OUTPUT:
Ajay   1
Ajay   2
Rahul  3
Neha   NULL

→ Unmatched rows → NULL

SELECT s.name, o.id
FROM students s
LEFT JOIN orders o
ON s.id = o.student_id
WHERE o.student_id IS NULL;

======================================================================
2.2) RIGHT JOIN (RIGHT OUTER JOIN)
======================================================================

DEFINITION:
Returns ALL rows from RIGHT table + matching from left.
👉 If no match → NULL values from right table
----------------------------------------------------------------------
EXAMPLE
----------------------------------------------------------------------
SELECT s.name, o.id
FROM students s
RIGHT JOIN orders o
ON s.id = o.student_id;

→ OUTPUT:
Ajay   1
Ajay   2
Rahul  3

→ All orders preserved
======================================================================
2.3) FULL OUTER JOIN (WORKAROUND)
======================================================================

DEFINITION:
Returns ALL rows from BOTH tables (matched + unmatched)

----------------------------------------------------------------------
MYSQL DOES NOT SUPPORT FULL OUTER JOIN DIRECTLY
----------------------------------------------------------------------

✔ USING UNION (REMOVES DUPLICATES)
----------------------------------------------------------------------
SELECT s.name, o.id
FROM students s
LEFT JOIN orders o
ON s.id = o.student_id

UNION

SELECT s.name, o.id
FROM students s
RIGHT JOIN orders o
ON s.id = o.student_id;

→ Removes duplicate rows automatically


----------------------------------------------------------------------
✔ USING UNION ALL (KEEPS DUPLICATES)
----------------------------------------------------------------------
SELECT s.name, o.id
FROM students s
LEFT JOIN orders o
ON s.id = o.student_id

UNION ALL

SELECT s.name, o.id
FROM students s
RIGHT JOIN orders o
ON s.id = o.student_id;

→ Keeps duplicate rows

----------------------------------------------------------------------
DIFFERENCE: UNION vs UNION ALL
----------------------------------------------------------------------

UNION:
• Removes duplicate rows
• Slightly slower (extra processing)

UNION ALL:
• Keeps duplicates
• Faster (no duplicate check)

----------------------------------------------------------------------
IMPORTANT NOTE
----------------------------------------------------------------------
• FULL JOIN via UNION → better (no duplicates)
• UNION ALL → may show duplicate matching rows twice

======================================================================
IMPORTANT OUTER JOIN NOTE
----------------------------------------------------------------------

NULL VALUES:
→ Non-matching rows filled with NULL

----------------------------------------------------------------------
COMMON MISTAKE (VERY IMPORTANT)
----------------------------------------------------------------------

SELECT s.name, o.id
FROM students s
LEFT JOIN orders o
ON s.id = o.student_id
WHERE o.id IS NOT NULL;

→ Acts like INNER JOIN ❌

======================================================================
3) CROSS JOIN
======================================================================

DEFINITION:
Returns all possible combinations of rows from both tables (Cartesian product)

----------------------------------------------------------------------
EXAMPLE:lets suppose we have 3 students and 3 orders, then...
----------------------------------------------------------------------
SELECT s.name, o.id
FROM students s
CROSS JOIN orders o;

→ OUTPUT:
3 students × 3 orders = 9 rows

Note: cross join will create huge dataset, so avoid using it.
only use it maybe while testing .
======================================================================
IMPORTANT CONCEPTS
----------------------------------------------------------------------

ON CLAUSE:
→ Defines join condition

----------------------------------------------------------------------
USING CLAUSE
----------------------------------------------------------------------
SELECT * FROM students
JOIN orders USING (id);

→ Shortcut when column names same

----------------------------------------------------------------------
SELF JOIN
----------------------------------------------------------------------
SELECT e.name, m.name AS manager
FROM employees e
JOIN employees m
ON e.manager_id = m.id;

----------------------------------------------------------------------
JOIN + WHERE
----------------------------------------------------------------------
SELECT s.name, o.id
FROM students s
JOIN orders o
ON s.id = o.student_id
WHERE s.name = 'Ajay';

----------------------------------------------------------------------
JOIN + GROUP BY
----------------------------------------------------------------------
SELECT s.name, COUNT(o.id)
FROM students s
LEFT JOIN orders o
ON s.id = o.student_id
GROUP BY s.name;

======================================================================
EDGE CASES
----------------------------------------------------------------------

1) NO MATCH
----------------------------------------------------------------------
👉 If no matches are found in an INNER JOIN → result is EMPTY (0 rows)LEFT JOIN → NULL values

----------------------------------------------------------------------
2) DUPLICATES
----------------------------------------------------------------------
Multiple matches → duplicate rows

----------------------------------------------------------------------
3) NULL VALUES
----------------------------------------------------------------------
NULL does not match in JOIN condition

----------------------------------------------------------------------
4) MISSING ON CONDITION
----------------------------------------------------------------------
→ Acts like CROSS JOIN (huge data ❌)

======================================================================
COMMON MISTAKES
----------------------------------------------------------------------

❌ Forgetting ON condition
❌ Using wrong join type
❌ Confusing LEFT vs RIGHT
❌ Not handling NULLs

======================================================================
JOIN SUMMARY
----------------------------------------------------------------------

JOIN TYPE     RETURNS
------------------------------------------
INNER         Matching rows only
LEFT          All left + matched right
RIGHT         All right + matched left
FULL          All rows both sides
CROSS         All combinations

======================================================================
JOIN vs SUBQUERY (ADVANTAGE OF JOIN)
======================================================================

• JOIN is faster and more efficient for large datasets  
• JOIN can directly display columns from multiple tables in one result  

• Subquery can also use multiple tables internally,
  but its result is passed to outer query (as value/set),
  not shown side-by-side like JOIN  

----------------------------------------------------------------------
FINAL UNDERSTANDING
----------------------------------------------------------------------
JOIN     → combine + display multiple tables together  
SUBQUERY → compute result, then use it in main query  

======================================================================
NOTES
----------------------------------------------------------------------
1) Joins are core of SQL
2) Mostly used: INNER + LEFT JOIN
3) Based on keys (PK ↔ FK)
4) Essential for real-world queries

======================================================================
MYSQL VIEWS 
======================================================================

DEFINITION:
👉 A VIEW is a virtual table created from a SQL query.
It does NOT store data itself — it stores the query.

🔹 Simple definition- A view is a saved SELECT query that behaves like a table.
----------------------------------------------------------------------
WHY VIEWS ARE USED
----------------------------------------------------------------------
• Simplify complex queries, instead of writing complex queries again n again, just create them once and call them using view.
• Improve security (hide not required columns)
• Reuse logic
• Provide abstraction

----------------------------------------------------------------------
SYNTAX 
----------------------------------------------------------------------
CREATE VIEW view_name AS
SELECT column1, column2
FROM table_name
WHERE condition;

----------------------------------------------------------------------
EXAMPLE
----------------------------------------------------------------------
CREATE VIEW student_view AS
SELECT name, age
FROM student
WHERE age > 18;

----------------------------------------------------------------------
USING VIEW
----------------------------------------------------------------------
SELECT * FROM student_view;

→ Works like a normal table

----------------------------------------------------------------------
VIEW WITH JOIN
----------------------------------------------------------------------
CREATE VIEW order_details AS
SELECT c.name, o.order_date
FROM customers c
JOIN orders o
ON c.customer_id = o.customer_id;


======================================================================
CONDITION ON VIEW
======================================================================

CREATE VIEW IND_emps AS
SELECT name, salary FROM employees
WHERE country = "India";

SELECT * FROM IND_emps WHERE salary > 5000;

----------------------------------------------------------------------
WHAT HAPPENS
----------------------------------------------------------------------

STEP 1:
View stores:
→ employees where country = 'India'

STEP 2:
Query on view adds condition:
→ salary > 5000

----------------------------------------------------------------------
INTERNAL EXECUTION
----------------------------------------------------------------------

SELECT name, salary
FROM employees
WHERE country = 'India' AND salary > 5000;

----------------------------------------------------------------------
KEY POINT
----------------------------------------------------------------------

• View condition + query condition are combined ✔  
• Acts like filtering on original table  

----------------------------------------------------------------------
FINAL RESULT
----------------------------------------------------------------------

→ Indian employees with salary > 5000

======================================================================
----------------------------------------------------------------------
UPDATABLE VIEW
----------------------------------------------------------------------
✔ Some views can be updated

Example:
UPDATE student_view
SET age = 25
WHERE name = 'Ajay';

Note: not a good practice to update view, instead update the root table or data.

----------------------------------------------------------------------
WHEN VIEW IS NOT UPDATABLE
----------------------------------------------------------------------

❌ Contains JOIN  
❌ Contains GROUP BY  
❌ Contains DISTINCT  
❌ Contains aggregate functions  

→ Then view becomes READ-ONLY

----------------------------------------------------------------------
DROP VIEW
----------------------------------------------------------------------
DROP VIEW view_name;

----------------------------------------------------------------------
REPLACE VIEW // it will override the old logic
----------------------------------------------------------------------
CREATE OR REPLACE VIEW view_name AS
SELECT ...;

example:
CREATE OR REPLACE VIEW hr_emps AS
SELECT name , salary, FROM employees
WHERE salary>4000;

----------------------------------------------------------------------
ADVANTAGES
----------------------------------------------------------------------

✔ Simplifies complex queries  
✔ Enhances security 
✔ Reusable logic  
✔ Cleaner code  

----------------------------------------------------------------------
DISADVANTAGES
----------------------------------------------------------------------

❌ Performance can be slower  
❌ Not always updatable  
❌ Depends on base tables  

----------------------------------------------------------------------
EDGE CASES
----------------------------------------------------------------------

1) BASE TABLE CHANGED
----------------------------------------------------------------------
→ View may break

----------------------------------------------------------------------
2) NO DATA STORAGE
----------------------------------------------------------------------
→ Always fetches from original table

----------------------------------------------------------------------
3) CASCADE EFFECT
----------------------------------------------------------------------
→ Dropping base table affects view

----------------------------------------------------------------------
NOTES
----------------------------------------------------------------------
1) View = virtual table  
2) Stores query, not data  
3) Useful in real-world systems  
4) Often used for security & abstraction  

======================================================================
SELECT vs VIEW (KEY DIFFERENCE)
======================================================================

SELECT:
----------------------------------------------------------------------
• Runs query every time you write it  
• Temporary (not saved)  
• You must write full query again  

Example:
SELECT name, age FROM student WHERE age > 18;

----------------------------------------------------------------------
VIEW:
----------------------------------------------------------------------
• Saved query (stored in database)  
• Reusable anytime  
• Acts like a virtual table  

Example:
CREATE VIEW student_view AS
SELECT name, age FROM student WHERE age > 18;

Then:
SELECT * FROM student_view;

----------------------------------------------------------------------
CORE DIFFERENCE
----------------------------------------------------------------------

SELECT → one-time query execution  
VIEW   → saved query you can reuse  

----------------------------------------------------------------------
WHY VIEW IS BETTER
----------------------------------------------------------------------

1) Avoid writing same query again  
2) Hide complex logic  
3) Provide security (hide columns)  
4) Cleaner and readable code  

----------------------------------------------------------------------
REAL-LIFE ANALOGY
----------------------------------------------------------------------

SELECT → solving math problem manually each time  
VIEW   → saving the answer/formula and reusing it
