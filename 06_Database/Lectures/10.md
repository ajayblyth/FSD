==================== STORED ROUTINES (MySQL) ====================
• Stored routines are reusable SQL blocks stored in the database.
• Used to reduce code repetition, improve performance, and centralize logic.

NOTE (Precompiled)
--------------------------------------------------------------------------------
• Precompiled means SQL is already parsed (validated for correctness) and optimized.
• Executes faster on repeated calls without re-parsing or re-optimizing.

TYPES OF STORED ROUTINES
--------------------------------------------------------------------------------

1) STORED PROCEDURE
--------------------------------------------------------------------------------
• Precompiled SQL routine used to perform operations like INSERT, UPDATE, DELETE,
  or complex business logic.
• Can return zero, one, or multiple values using OUT parameters.
• Called using CALL statement.
• Used mainly for executing tasks and data-modifying operations.

2) STORED FUNCTION
--------------------------------------------------------------------------------
• Precompiled SQL routine that MUST return exactly one value using RETURN.
• Used inside SQL queries (SELECT, WHERE, ORDER BY, etc.) like built-in functions.
• Mainly used for calculations or deriving values (does not modify data).

3) TRIGGER
--------------------------------------------------------------------------------
• Not called manually.
• Automatically executes when an event occurs on a table.
• Events: INSERT, UPDATE, DELETE.

WHY USE STORED ROUTINES
--------------------------------------------------------------------------------
• Reusability      → Write once, use many times
• Performance      → Precompiled (faster execution)
• Security         → Restrict direct table access
• Maintainability  → Business logic stays inside database

KEY CONCEPTS
--------------------------------------------------------------------------------
OPERATIONS (Stored Procedure context)
• Actions that MODIFY database data.
• Examples: INSERT, UPDATE, DELETE, MERGE.

CALCULATIONS (Stored Function context)
• Compute/derive values WITHOUT modifying data.
• Examples: salary, tax, discount, ranking.

KEY DIFFERENCE
--------------------------------------------------------------------------------
• Stored Procedure → Can perform operations (data change) + calculations if needed.
• Stored Function  → Used for calculations, MUST return a value, should NOT modify data.
• Trigger          → Automatically executes on table events (no manual call).

EXECUTION BEHAVIOR
--------------------------------------------------------------------------------
• Stored Procedure / Function → Manually executed
• Trigger                    → Automatically executed

DELIMITER NOTE
--------------------------------------------------------------------------------
• Required for multi-line SQL blocks (BEGIN...END).
• Applies to procedures, functions, and triggers.

INTERVIEW LINE 🚀
--------------------------------------------------------------------------------
• Triggers are event-driven database objects that execute automatically on table events.

1) STORED PROCEDURE
===============================================================

-- Create Procedure
DELIMITER $$ 

CREATE PROCEDURE getAllEmployees()
BEGIN
    SELECT * FROM employees;
END $$

DELIMITER ;

-- Call Procedure
CALL getAllEmployees();

Note: wrap your procedure in delimitter.

DELIMITER Theory:
• DELIMITER is used to change the default statement terminator (;) temporarily.
• Needed because stored routines contain multiple SQL statements using ;
• Example: DELIMITER $$ allows writing full procedure, ending with $$ instead of ;
• After definition, reset back using DELIMITER ;

Without DELIMITER:

• MySQL will treat the first ; inside BEGIN...END as end of query
• Procedure/function will break or give syntax error
• It won’t understand full block as one unit

================ TOTAL SALARY BY DEPARTMENT ====================

-- Assumed Table:
-- employees(id, name, salary, dept)

DELIMITER $$

CREATE PROCEDURE total_salary_by_dept(
    IN dept_name VARCHAR(20),
    OUT total_salary INT
)
BEGIN
    SELECT SUM(salary)
    INTO total_salary
    FROM employees
    WHERE dept = dept_name;   -- ensure column name matches table
END $$

DELIMITER ;

---------------------------------------------------------------
HOW IT WORKS
---------------------------------------------------------------

• IN dept_name     → Input value (e.g., 'Sales')
• OUT total_salary → Stores result (SUM of salaries)
• SELECT...INTO    → Calculates + assigns value to OUT variable

---------------------------------------------------------------
HOW TO CALL
---------------------------------------------------------------

-- Step 1: Call procedure
CALL total_salary_by_dept('Sales', @dept_total);

-- Step 2: View result
SELECT @dept_total;


---------------------------------------------------------------
IMPORTANT NOTES
---------------------------------------------------------------

• If no rows match → SUM returns NULL
• Use IFNULL(SUM(salary), 0) if you want 0 instead of NULL
• OUT variable must be user variable (@var)

2) STORED FUNCTION
===============================================================

-- Function must RETURN a value

DELIMITER $$

CREATE FUNCTION getBonus(salary INT)
RETURNS INT
DETERMINISTIC
BEGIN
    RETURN salary * 0.10;
END $$

DELIMITER ;

-- Call Function
SELECT getBonus(5000);



---------------------------------------------------------------
Concept:
• Functions are like "expressions"
• Used inside SELECT, WHERE, ORDER BY
• Must return exactly ONE value
• Cannot modify database (best practice: no INSERT/UPDATE)
---------------------------------------------------------------
==================== DETERMINISTIC vs NON-DETERMINISTIC ====================
1. What is DETERMINISTIC

👉 It tells MySQL whether the function always returns same output for same input

DETERMINISTIC     → Same input → Same output (always)
NOT DETERMINISTIC → Same input → Output may change

2. Examples
Deterministic
CREATE FUNCTION add_bonus(salary INT)
RETURNS INT
DETERMINISTIC
RETURN salary * 0.10;

👉 Input 1000 → always 100

Not Deterministic
CREATE FUNCTION get_today()
RETURNS DATE
NOT DETERMINISTIC
RETURN CURDATE();

👉 Output changes every day
DETERMINISTIC:

• Same input → ALWAYS same output
• No dependency on external factors
• Example: salary * 0.10

❓ 1. Why NO DELIMITER and BEGIN...END here?

✅ Reason:

• This is a single statement function
• MySQL allows direct RETURN without BEGIN...END

When DO you need BEGIN...END + DELIMITER?

👉 When function has multiple statements

CREATE FUNCTION getBonus(salary INT)
RETURNS INT
DETERMINISTIC
BEGIN
    RETURN salary * 0.10;
END;

==================== FUNCTION: CALCULATE PRICE ==================

DELIMITER $$

CREATE FUNCTION cal_price(
    item_price INT,
    tax_rate INT
)
RETURNS INT
DETERMINISTIC
BEGIN
    DECLARE final_price INT;

    SET final_price = item_price + (item_price * tax_rate / 100);

    RETURN final_price;
END $$

DELIMITER ;

---------------------------------------------------------------
HOW TO USE
---------------------------------------------------------------

SELECT cal_price(1000, 10);   -- Output: 1100
SELECT cal_price(500, 18);    -- Output: 590


---------------------------------------------------------------
LOGIC EXPLANATION
---------------------------------------------------------------

Function Purpose:
• Calculates final price after adding tax

Parameters:
• item_price → base price of item
• tax_rate   → tax percentage

Return Type:
• RETURNS INT → final result is integer

DETERMINISTIC:
• Same input → always same output
• Example: (1000,10) → always 1100

Calculation:
• final_price = item_price + (item_price * tax_rate / 100)


---------------------------------------------------------------
IMPORTANT NOTES
---------------------------------------------------------------

• Uses local variable (DECLARE)
• RETURN is mandatory in function
• Can be used inside SELECT queries
• Best for reusable calculations

==================== FUNCTION: FIRST 3 CHAR UPPERCASE ==================

DELIMITER $$

CREATE FUNCTION first3_upper(name VARCHAR(50))
RETURNS VARCHAR(10)
DETERMINISTIC
BEGIN
    RETURN UPPER(SUBSTRING(name, 1, 3));
END $$

DELIMITER ;

---------------------------------------------------------------
HOW TO USE
---------------------------------------------------------------

SELECT first3_upper('ajay');      -- Output: AJA
SELECT first3_upper('rahul');     -- Output: RAH
SELECT first3_upper('john');      -- Output: JOH


---------------------------------------------------------------
LOGIC
---------------------------------------------------------------

• SUBSTRING(name, 1, 3) → gets first 3 characters
• UPPER()               → converts to uppercase
• Combined → UPPER(SUBSTRING(...))

---------------------------------------------------------------
NOTES
---------------------------------------------------------------

• Works even if name is longer
• If name < 3 chars → returns available chars only
---------------------------------------------------------------

---------------------------------------------------------------
NON-DETERMINISTIC:

• Same input → MAY give different output
• Depends on time, random values, or database state

Examples:
• NOW()        → current date/time
• RAND()       → random number
• SELECT COUNT(*) FROM table → data may change

CREATE FUNCTION getCurrentTime()
RETURNS DATETIME
NOT DETERMINISTIC
BEGIN
    RETURN NOW();
END;


---------------------------------------------------------------
INTERVIEW POINT:

• DETERMINISTIC → predictable, optimizable
• NOT DETERMINISTIC → unpredictable, depends on external state

Feature        | View                     | Procedure                | Function
---------------|--------------------------|--------------------------|-------------------------

Storage        | Stored query             | Stored in DB             | Stored in DB

==================== TRIGGERS (MySQL) ====================

Definition:
A trigger is a block of SQL that automatically executes
when an event (INSERT, UPDATE, DELETE) occurs on a table.

---------------------------------------------------------------
TYPES / TIMING
---------------------------------------------------------------

• BEFORE INSERT
• AFTER INSERT
• BEFORE UPDATE
• AFTER UPDATE
• BEFORE DELETE
• AFTER DELETE

---------------------------------------------------------------
BASIC SYNTAX
---------------------------------------------------------------

DELIMITER $$

CREATE TRIGGER trigger_name
BEFORE/AFTER INSERT/UPDATE/DELETE ON table_name
FOR EACH ROW
BEGIN
    -- SQL logic
END $$

DELIMITER ;

---------------------------------------------------------------
EXAMPLE 1: BEFORE INSERT (DEFAULT VALUE)
---------------------------------------------------------------

D================ EXAMPLE: BEFORE INSERT (VALIDATION ERROR) ================

DELIMITER $$

CREATE TRIGGER before_emp_insert
BEFORE INSERT ON employees
FOR EACH ROW
BEGIN
    IF NEW.salary IS NULL THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Salary cannot be NULL';
    END IF;
END $$

DELIMITER ;

Note:
IF is a control structure, so it must be closed with END IF.
NEW refers to the new row being inserted or updated in a table.
• Used inside triggers to access incoming values.

SIGNAL SQLSTATE is used to raise custom errors in MySQL,
and MESSAGE_TEXT defines the error message shown to the user.

---------------------------------------------------------------
HOW IT WORKS
---------------------------------------------------------------

• BEFORE INSERT → runs before data is inserted
• IF salary is NULL → trigger throws error
• INSERT operation is stopped

---------------------------------------------------------------
RESULT
---------------------------------------------------------------

INSERT INTO employees(name, salary)
VALUES('Ajay', NULL);

→ ERROR: Salary cannot be NULL

---------------------------------------------------------------
CONCEPT
---------------------------------------------------------------

• SIGNAL SQLSTATE '45000' → used to raise custom error
• Prevents invalid data from entering table
• Useful for strict validation rules
---------------------------------------------------------------

---------------------------------------------------------------
EXAMPLE 2: AFTER INSERT (LOGGING)
---------------------------------------------------------------

-- orders_log(order_id, log_msg)

DELIMITER $$

CREATE TRIGGER insert_orders
AFTER INSERT ON orders
FOR EACH ROW
BEGIN
    INSERT INTO orders_log(order_id, log_msg)
    VALUES(
        NEW.order_id,
        CONCAT('ORDER ', NEW.order_id, ' created')
    );
END $$

DELIMITER ;

---------------------------------------------------------------
FLOW (AFTER INSERT)
---------------------------------------------------------------

1) Insert into orders
2) Trigger fires automatically
3) Log inserted into orders_log

---------------------------------------------------------------
EXAMPLE 3: AFTER INSERT (EMP LOG)
---------------------------------------------------------------

-- emp_log(id, message)

DELIMITER $$

CREATE TRIGGER after_emp_insert
AFTER INSERT ON employees
FOR EACH ROW
BEGIN
    INSERT INTO emp_log(message)
    VALUES(CONCAT('Employee added: ', NEW.name));
END $$

DELIMITER ;

---------------------------------------------------------------
EXAMPLE 4: BEFORE UPDATE (VALIDATION)
---------------------------------------------------------------

DELIMITER $$

CREATE TRIGGER before_emp_update
BEFORE UPDATE ON employees
FOR EACH ROW
BEGIN
    IF NEW.salary < 0 THEN
        SET NEW.salary = OLD.salary;
    END IF;
END $$

DELIMITER ;

---------------------------------------------------------------
CUSTOM ERROR (VALIDATION)
---------------------------------------------------------------

SIGNAL SQLSTATE '45000'
SET MESSAGE_TEXT = 'Error message';

Example:
IF NEW.salary < 0 THEN
    SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = 'Salary cannot be negative';
END IF;

---------------------------------------------------------------
KEY CONCEPTS
---------------------------------------------------------------

• NEW → new row values (INSERT/UPDATE)
• OLD → old row values (UPDATE/DELETE)
• FOR EACH ROW → runs per affected row
• Automatic execution (no CALL needed)

---------------------------------------------------------------
IMPORTANT POINTS
---------------------------------------------------------------

• Cannot be called manually
• Only 1 trigger per event per table (MySQL)
• Used for validation, logging, auditing
• May affect performance if overused

---------------------------------------------------------------
SHOW / DELETE
---------------------------------------------------------------

SHOW TRIGGERS;

SHOW CREATE TRIGGER before_emp_insert;

DROP TRIGGER before_emp_insert;

---------------------------------------------------------------
INTERVIEW SUMMARY
---------------------------------------------------------------

• Trigger = automatic DB-level logic
• BEFORE → validation / modification
• AFTER  → logging / audit
• Ensures rules are always enforced

==================== WHY USE TRIGGERS ====================

Purpose:
Triggers automate actions in response to data changes.

---------------------------------------------------------------
USE CASES
---------------------------------------------------------------

1) DATA VALIDATION
• Ensure correct data before insert/update
• Example: prevent negative salary

2) DEFAULT VALUES
• Automatically set values if not provided
• Example: set salary = 10000 if NULL

3) AUDIT / LOGGING
• Track changes in another table
• Example: log "employee added"

4) MAINTAIN CONSISTENCY
• Keep related tables in sync
• Example: update total count in another table

5) SECURITY / RESTRICTIONS
• Prevent unwanted operations
• Example: block delete if condition fails

6) AUTOMATION
• No need to manually write extra queries
• Runs automatically on INSERT/UPDATE/DELETE

---------------------------------------------------------------
WHEN TO USE
---------------------------------------------------------------

• When logic must ALWAYS execute automatically
• When you want DB-level control (not app-level)
• For auditing, validation, consistency

---------------------------------------------------------------
INTERVIEW LINE
---------------------------------------------------------------

"Triggers are used to automatically enforce rules,
maintain consistency, and log changes at database level."

==================== TRIGGERS ASSIGNMENT ====================

TABLE ASSUMPTION:
-- orders(order_id, price, status)
-- orders_log(order_id, message)

===============================================================
Q1) AFTER UPDATE → LOG PRICE CHANGE
===============================================================

Requirement:
• If price is updated → log old price and new price

DELIMITER $$

CREATE TRIGGER after_price_update
AFTER UPDATE ON orders
FOR EACH ROW
BEGIN
    IF OLD.price <> NEW.price THEN
        INSERT INTO orders_log(order_id, message)
        VALUES(
            NEW.order_id,
            CONCAT('Price changed from ', OLD.price, ' to ', NEW.price)
        );
    END IF;
END $$

DELIMITER ;


===============================================================
Q2) BEFORE DELETE → PREVENT DELETE IF SHIPPED/DELIVERED
===============================================================

Requirement:
• If status = 'shipped' or 'delivered' → do NOT allow delete

DELIMITER $$

CREATE TRIGGER before_order_delete
BEFORE DELETE ON orders
FOR EACH ROW
BEGIN
    IF OLD.status IN ('shipped', 'delivered') THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Cannot delete shipped or delivered orders';
    END IF;
END $$

DELIMITER ;


===============================================================
Q3) AFTER DELETE → LOG DELETION
===============================================================

Requirement:
• If order is deleted → log that deletion

DELIMITER $$

CREATE TRIGGER after_order_delete
AFTER DELETE ON orders
FOR EACH ROW
BEGIN
    INSERT INTO orders_log(order_id, message)
    VALUES(
        OLD.order_id,
        CONCAT('Order ', OLD.order_id, ' has been deleted')
    );
END $$

DELIMITER ;


===============================================================
INTERVIEW TAKEAWAY
===============================================================

• AFTER UPDATE → track changes (audit)
• BEFORE DELETE → restrict invalid operations
• AFTER DELETE → maintain deletion logs
